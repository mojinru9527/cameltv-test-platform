# 测试用例设计（功能 / 接口）

## Overview

输出任何「功能用例 / 测试用例 / 接口用例」时,**必须**遵循 `tests/test-case-standards/` 下的团队规范,而不是凭直觉随意编写。本 skill 把那套规范固化成可执行流程:**先分析、再设计、套检查点、按模板输出、保证覆盖率**。

**核心原则:用例集合必须 100% 覆盖需求功能点;每个需求点至少一条正面用例 + 一条负面用例。**

## 何时使用

- 用户说「输出功能用例 / 写测试用例 / 设计接口用例 / 补充用例」
- 需求评审后要为新功能/新接口设计用例
- 已有用例需要维护(新增、修改、删除过时用例)
- 用例评审前的自检

## 强制工作流程

**不要跳过任何一步直接堆用例。** 按顺序执行:

### 第 1 步:测试需求分析
- 找到需求来源:需求说明书 / PingCode 故事 / 接口文档。**信息不足时先向用户索取**,不要臆造。
- 拆出被测对象的全部**功能点**(单接口需逐一列出:接口说明、请求方式、URL、请求参数、返回数据)。
- 输出一个功能点清单,作为覆盖率基线。

### 第 2 步:业务流程分析
- 梳理**主流程(基本流)**、**备选流**、**异常流**、数据流向、关键判断条件、前置条件。
- 复杂模块用「场景法」:基本流 + 各备选流组合 = 一个个场景(见 `测试用例具体用法.md` 的 ATM 场景矩阵)。

### 第 3 步:套用检查点设计用例
按被测类型加载对应检查清单,逐条比对、不遗漏:

| 被测类型 | 必读检查清单 |
| --- | --- |
| 功能用例 | [functional-checklist.md](functional-checklist.md) + 源文档 `测试的具体体现.md`、`功能测试检查点【辅助】.md` |
| 接口/API 用例 | [api-checklist.md](api-checklist.md) + 源文档 `API接口测试方案.md` |
| 界面用例 | 源文档 `界面测试的方法要点.md`、`测试用例标准.md` 第 4.2 节 |

设计方法综合运用:**等价类划分、边界值分析、场景法、因果图、正交表、错误推测**(详见 `功能测试用例规范.md`「用例设计方法」)。

### 第 4 步:按标准模板输出
用例字段、模板与示例见 [case-template.md](case-template.md)。**每条用例必须包含**:用例标题、前提条件、明确的操作步骤(含输入数据)、可验证的预期结果。

- 预期结果不能只写界面提示,需含**显式验证手段**(如「数据库中查询记录已更新」)。
- 重要程度按 **P0 / P1 / P2** 标级(定义见模板文件)。

### 第 5 步:覆盖率自检
输出后,对照第 1 步功能点清单逐项确认:

- [ ] 每个需求功能点 ≥ 1 条用例(基本流覆盖率 100%)
- [ ] 每个需求点都有正面用例 + 负面用例
- [ ] 边界值、异常输入、并发/兼容/性能(按需)已覆盖
- [ ] 接口用例:入参校验 + 业务逻辑校验 + 返回值校验 三类齐全(必选项)

## 用例级别速查

- **P0**:关键路径、最常用功能、基本流、界面数据有效性校验 → 全过则功能稳定
- **P1**:不常用功能、异常流、边界值、异常数据 → 全过则可发布
- **P2**:GUI/界面显示/提示一致性、可用性、压力性能等 → 体验类

## Red Flags — 停下来重做

- 没看需求/接口文档就开始写用例 → 回到第 1 步
- 用例只有正面、没有负面/异常用例 → 覆盖不全
- 接口用例缺少返回值校验或业务逻辑校验 → 违反必选项
- 预期结果只写「成功」「报错」无具体判断标准 → 不可执行
- 不套检查清单、凭记忆写 → 必然遗漏

## 维护规则(修改已有用例时)

- 需求变更:同步修改用例,备注栏说明;**用例执行超 2/3 后原则上不再变更/新增需求**。
- 过时用例:置为废弃状态(不直接物理删除计划内用例)。
- 冗余用例:相同输入/输出的多条只保留一条。
- 新增用例:插入到对应功能模块最下方,备注说明。

详见 `功能测试用例规范.md`「用例的维护」、`缺陷管理制度.md`。
