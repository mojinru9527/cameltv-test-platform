# 体育测试平台 · 测试自动化平台建设方案（CI/CD）

> **顶层目标**：先用现有体育项目（CamelTv）把这套测试自动化平台跑通落地，再抽象成**可移植的测试平台**——其他项目能**一键部署**套用；并提供一个**前端页面**给测试同学日常使用。

## 平台架构总览（贯穿全文的设计原则）

为达成"可移植 + 一键部署"，整套平台从一开始就按**「内核 / 项目配置」分离**来建：

```
┌─────────────────────────────────────────────────────────┐
│  前端控制台 (Web UI)  —— 测试同学日常入口                  │
│  选项目 → 选环境(test/prod) → 触发测试 → 看报告/日志        │
└───────────────┬─────────────────────────────────────────┘
                │ REST API
┌───────────────▼─────────────────────────────────────────┐
│  平台内核 (与具体项目无关，跨项目复用)                      │
│  环境管理 · 任务编排 · API测试引擎 · UI自动化引擎 ·          │
│  造数 · 日志(ELK)接入 · 报告聚合 · CI/CD 触发              │
└───────────────┬─────────────────────────────────────────┘
                │ 读取
┌───────────────▼─────────────────────────────────────────┐
│  项目配置包 (每个项目一份，唯一需要改的东西)                 │
│  环境/域名/代理 · 接口基线(swagger) · 用例集 · 日志源 ·      │
│  数据库/缓存连接 · CI 流水线参数                            │
└─────────────────────────────────────────────────────────┘
```

**移植到新项目 = 复制一份"项目配置包" + 一键部署内核**，内核代码不动。现有 [test-platform/config/](test-platform/config/) 的"站点×环境合并"机制已经是这个思路的雏形，后续所有模块都遵循它。

## 1. 环境管理：测试 / 正式环境切换（平台地基）

**目标**：任一测试动作绑定到明确环境，沿用该环境的接口地址与日志源。

**现状** 🟡：`test-platform/config/` 已有环境合并体系，但真实环境信息待登记。

**已确认的环境模型** ✅：
- **本产品为「1 产品多镜像」，后端服务只有 1 套**，因此**不按站点拆分，只区分两个环境**：`test`（测试）/ `prod`（正式）。多个域名只是同一产品的镜像入口。
- **测试环境**（需**内网**访问，CI runner 必须在内网运行）：
  `camel-g5-test5 / g3-test3 / g3-test1 / g1-test5 / g2-test5` 等 `*.elelive.cn` 域名。
- **正式环境**（需 **vpn07 代理 tun 模式 + 全局流量**才能访问）：
  `www.camel1.tv / camell.to / camel2.live / camelscore.live / goals365.live / camellive.top`。

**要做的事：**
1. 配置收敛为单项目两环境：`config/environments/test.yaml`、`config/environments/prod.yaml`（域名/代理/DB/Redis/MQ/期望版本/日志源）。
2. 代理策略固化：`test` = 内网直连；`prod` = vpn07 tun + 全局流量（写入 `proxy` 字段 + 启动前置校验，缺代理直接 fail fast）。
3. 统一入口：所有命令 `--env test|prod`，让接口/日志/DB 三者随环境联动。

**产出物**：`test.yaml` / `prod.yaml` 两份环境配置；`tp config show --env test|prod` 可验证。

## 4. API 测试（实现方案：官方 Playwright MCP）🔴 核心新建

**目标**：以**官方 Playwright MCP** 做接口自动化，覆盖断言、DB/缓存核对、失败重试、定时任务。

**已确认** ✅：技术选型采用**官方 Playwright MCP**（非自研封装）。

**用例来源（两条路线，已确认优先级）** ✅：
- 🥇 **优先：swagger 生成** —— 拉各服务 swagger/OpenAPI 自动生成用例骨架。生成的用例需**备注来源 `source: swagger`**。
- 🥈 **补充：UI 捕获反推** —— 从 §5 UI 自动化运行时捕获的接口存档反推用例，按服务区分。生成的用例需**备注来源 `source: ui-capture`**。
- 两条路线对同一接口去重时，**以 swagger 版本为准**，UI 捕获仅用于补 swagger 未覆盖的接口。

**要做的事（逐子节点）：**

| 子项 | 落地做法 | 复用现状 |
| --- | --- | --- |
| swagger → 用例 | 拉 OpenAPI 生成骨架，标 `source: swagger` | 对齐 §3 |
| 测试数据 | 用例与造数解耦 | 复用 [data_factory](test-platform/tools/data_factory/) |
| 配置文件 | 接口/凭据随 `--env` 注入 | 复用 [config/](test-platform/config/) |
| 断言校验 | 状态码 + JSON 字段 + schema | 借 [api_diff](test-platform/tools/api_diff/) 字段比对 |
| MySQL / Redis | 接口后查 DB/Redis 验落库与缓存 | 复用 env 连接 |
| 失败机制 | 失败重试 + 自动采集 traceId 给 §7 | 与日志联动 |
| 定时任务 | 定时跑回归（CI 定时触发） | 接入 CI |
| 实现策略 | swagger 为主、UI 捕获补充，按服务区分并标注来源 | 依赖 §5 |

**产出物**：`tests/api-testing/` 下 Playwright MCP 用例工程（用例头部带 `source` 备注）+ 接口报告。

## 5. UI 自动化（midscene.js）🔴 新建

**已确认的合规策略** ✅：

**要做的事：**

2. 输入 = §3 标注的 **P0 功能用例** → 转 UI 自动化脚本，持续运行 + 出报告。

4. 环境绑定 `--env`。

**产出物**：UI 自动化用例 + 报告（汇入 §8）+ 接口流量存档。

## 11. 前端控制台（Web UI）🔴 新建 —— 测试同学的统一入口

**目标**：给测试同学一个网页，不用敲命令就能用整套平台。**这是"可移植平台"的门面**。

**核心功能（MVP）：**
1. **选项目 + 选环境**（test / prod）。
2. **一键触发**：环境探活 / API 回归 / UI 自动化 / 造数。
3. **看报告**：功能 / 接口 / UI 三类报告 + 趋势 + flaky（并入 §8）。
4. **看日志**：失败用例一键跳 ELK（§7）。
5. **任务历史**：每次运行的环境、版本、结果、耗时。

**技术建议**：前端 React/Vue + 后端 FastAPI（包一层 REST 调用现有 `tp` CLI 引擎）。后端即 §架构图 的"平台内核 API 层"。

**产出物**：可访问的 Web 控制台 + REST API 服务。

## 12. 可移植 & 一键部署（最终形态）🔴

**目标**：平台与项目解耦，新项目**复制一份配置包 + 一键部署**即可套用。

**要做的事：**
1. **内核/项目分离**：把"与项目无关的引擎"（环境管理、API/UI 引擎、造数、ELK、报告、前端）与"项目配置包"（环境域名/代理、swagger、用例、日志源、DB/Redis、CI 参数）彻底分开。现有 [config/](test-platform/config/) 已是雏形。
2. **容器化一键部署**：用 `docker-compose`（前端 + 内核 API + WireMock + sqlite/PG + 可选 ELK 代理）一条命令起全套。
3. **项目接入向导**：`tp init-project <name>` 生成一份空白项目配置包模板，测试同学填域名/凭据/swagger 即可接入新项目。
4. **CamelTv 作为第一个"样板项目"**验证整套流程，跑通后再抽模板。

**产出物**：`docker-compose.yml` 一键部署 + 项目配置包模板 + 接入文档。

## 15. CI/CD 主线流程

```
触发(push / 定时)  [CI runner 在内网；prod 任务起 vpn07 代理]
  → tp envcheck            环境探活，红则 fail fast（§1）
  → tp datafactory         按需造数（§4 数据）
  → API 测试(Playwright MCP) 接口回归 + DB/Redis 校验（§4，swagger 优先）
  → UI 自动化(midscene)     P0 场景 + 捕获接口存档（§5，主跑 test）
  → 失败 → ELK + traceId    自动定位（§7）
  → tp report ingest        三类报告入库（§8）
  → 前端控制台展示结果        测试同学查看（§11）
  → (上线前) capture+apidiff 生产流量回放验回归
```
