## 1. 部署 / 升级步骤（务必按序执行）

> ⚠️ 顺序不可颠倒：**先迁移库、再开 Worker**。

1. **停用证据 Worker**（升级期间避免半态任务）
   在后端 `.env` 设置：
   ```
   LANHU_EVIDENCE_WORKER_ENABLED=false
   ```
2. **应用数据库迁移**（新增质量/恢复相关列与索引，非破坏性）
   ```bash
   cd test-platform-v2/backend
   ./.venv/Scripts/python.exe -m alembic upgrade head
   ./.venv/Scripts/python.exe -m alembic current   # 应指向 20260714_lanhu_pg_reconcile (head)
   ./.venv/Scripts/python.exe -m alembic heads      # 只应有一个 head
   ```
   - SQLite：通过 `batch_alter_table` 自动补列，直接执行即可。
   - PostgreSQL：同一条迁移链会补齐历史部署漏建的复合索引，并为非空质量/恢复列设置 server default，可重复执行（幂等）。
3. **跑自动化门禁**（见附录 A），全绿后再继续。
4. **启用证据 Worker**：
   ```
   LANHU_EVIDENCE_WORKER_ENABLED=true
   ```
   重启后端，确认 `/health` 正常，创建一个证据任务，确认**恰好一个** Worker 认领且无类型/默认值/事务错误。

## 2. 新增环境变量

| 变量 | 默认 | 说明 |
|------|------|------|
| `LANHU_EVIDENCE_WORKER_ENABLED` | `true` | 是否在调度 Worker 中轮询执行证据任务 |
| `LANHU_EVIDENCE_MAX_CONCURRENT` | `1` | 同时执行的证据任务上限（线程信号量） |
| `LANHU_EVIDENCE_STALE_AFTER_SECONDS` | `600` | 心跳超过该秒数未更新的 `running` 任务判定为丢失（`worker_lost`） |

> 心跳间隔约为 `stale_after / 20`（默认约 30s），正常运行任务每窗口写入约 20 次心跳，不会被误杀。

## 3. 蓝湖证据包导入 — 标准操作流程

证据导入现为**门禁状态机**：先采证 → 生成可审计质量报告 → 每页需 OCR 成功或人工复核 → 才允许正式导入到 需求 / 知识库(RAG) / Wiki。

### 3.1 创建任务
- 入口：知识库页 → 蓝湖证据任务抽屉（`LanhuEvidenceJobDrawer`）。
- 选项：`capture_all_pages`、`include_word`、`include_json`，以及三个下游导入开关（需求/知识库/Wiki）。**这些选项会被真实持久化并执行**，不再被忽略。
- 创建后仅落一条 `pending` 任务并返回 HTTP 200，由 Worker 异步认领执行。

### 3.2 任务状态语义
| 状态 | 含义 | 能否导入 |
|------|------|----------|
| `success` | 每页均有截图、无截断、合并文本非空，且 OCR 成功或已人工批准 | ✅ 可导入 |
| `success_with_warnings` | 有页面缺 OCR/复核、截断或空文本 | ❌ 导入返回 **HTTP 409** |
| `failed` | 零截图（`error_message="No Lanhu page screenshot was captured"`）或 `worker_lost` | ❌ 不产出任何文档/下游记录 |

### 3.3 页面人工复核（OCR 不可用时的合规通道）
当某页 OCR 不可用，需人工复核放行：
- 接口：`POST /api/v1/lanhu-evidence/pages/{page_id}/review`
  ```json
  { "approved": true, "comment": "至少 3 字的审计说明" }
  ```
- 权限点：`lanhu_evidence:review`（本次新增，已在 seed 注入超级管理员）。
- 约束：只允许对**已有截图且合并文本非空**的页面批准；每次复核后自动重算父任务质量。
- 只有当没有任何页面缺采集/缺文本/被截断/缺 OCR 或复核时，`import_ready` 才会变 `true`。

### 3.4 正式导入
- 接口：`POST /api/v1/lanhu-evidence/jobs/{job_id}/import`，body 指定 `import_to_requirement/knowledge/wiki`。
- 门禁：服务层要求 `status=="success"` 且 `quality.import_ready==true`，否则 **409**。
- 三条下游导入统一经 `import_service`，只执行被持久化且为 true 的请求标志。

### 3.5 产物下载（不再暴露服务器路径）
- 列出资产：`GET /api/v1/lanhu-evidence/jobs/{job_id}/assets` → 返回资产列表（含 `asset_id`、类型、sha256）。
- 下载：`GET /api/v1/lanhu-evidence/assets/{asset_id}`（唯一下载路由，带项目+路径校验）。
- 前端仅按 `asset_id` 渲染下载链接，**不再渲染 `word_path`/`json_path`**。
- 校验：Word 章节数 与 JSON 页数 应等于发现页数。

### 3.6 中断恢复与重试
- `running` 任务心跳超时 → 自动置 `failed` + `error_message="worker_lost"`。
- 重试：`POST /api/v1/lanhu-evidence/jobs/{job_id}/retry` → 创建**全新** `pending` 任务：
  - `parent_job_id=旧ID`、`attempt_no=旧+1`、独立目录 `.../{new_id}/attempt-{n}`。
  - **绝不改动**原任务/页面/资产/已导出证据，也不混用新旧产物。

## 4. Agent 队列写入可靠性

- `enqueue` 复用调用方 DB 会话（不再内部新开 `SessionLocal()`），对 `database is locked` 最多重试 3 次（指数退避）。
- 重试耗尽 → 抛 `QueueWriteBusy`，路由层 `POST /api/v1/agents/run/{agent_type}` 返回**受控的 HTTP 503**（"Agent queue is temporarily busy; retry shortly"），不再抛裸 SQL 栈的 500。
- 客户端遇 503 应稍后重试。

## 5. 项目隔离与权限

- **API 测试资产**：`/apitest/cases/generate` 与 `/cases/batch-generate` 按当前项目校验 endpoint，跨项目返回 404/跳过。
- **删除路由**：`DELETE /apitest/services/{id}`、`DELETE /apitest/endpoints/{id}` 均按项目作用域；被引用时返回 409。
  > ⚠️ 实际所需权限点为 **`apitest:asset_manage`**（非实施方案文字中的 `apitest:manage`）。分配角色时以此为准。
- **Wiki lint 覆盖**：`project_id_override` 仅**超级管理员**可用，其余账号返回 403。`is_super` 现仅由全局（`project_id=0`）角色派生——seed 已确认 admin 的超管角色挂在 `project_id=0`。

## 6. UI 自动化产物隔离

- Playwright 执行改用 `threading.BoundedSemaphore(MAX_CONCURRENT)`，在 `finally` 中释放；并发满时该 run 置回 `pending` 等待下轮认领。
- 通过 `--output <artifact_dir>` 与 `PLAYWRIGHT_JSON_OUTPUT_NAME=<artifact_dir>/report.json` 让产物天然隔离；**删除**了对共享 `tests/playwright/test-results` 的拷贝扫描。
- 每个 UI run 的产物接口只返回自己 `storage/ui-runs/{run_id}` 目录下的文件，杜绝跨 run 串档。

## 7. 回滚方案

- **应用层回滚**：设 `LANHU_EVIDENCE_WORKER_ENABLED=false` 即停用证据 Worker。**不要**降级迁移——复核/心跳数据向后兼容，降级会丢列。
- **历史 success 任务**：若 `ocr_pages < total_pages`，用一次性脚本将其 `quality_json.import_ready=false`，**不要删文件**。
- **紧急导入**：走页面复核批准（带书面理由），**禁止**直接 SQL 改 `status`。

## 8. 常见问题（FAQ）

- **Q：任务一直 `success_with_warnings` 无法导入？**
  A：查看质量报告的 `pages_missing_capture / pages_truncated / pages_missing_text / pages_missing_ocr_review` 列表，逐页补采或人工复核，直到 `import_ready=true`。
- **Q：调用 Agent 触发接口偶发 503？**
  A：SQLite 短时写锁，稍后重试即可；持续 503 请调大并发或迁移 PostgreSQL。
- **Q：任务卡在 `running`？**
  A：等待超过 `LANHU_EVIDENCE_STALE_AFTER_SECONDS` 会自动判 `worker_lost`，再走 `/retry`。
- **Q：下载链接 404？**
  A：确认使用 `asset_id` 走 `GET /assets/{asset_id}`，且当前用户属于该资产所在项目。
