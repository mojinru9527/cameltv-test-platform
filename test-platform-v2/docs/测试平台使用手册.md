# CamelTv 测试平台使用手册

版本：2.6
更新日期：2026-07-15
适用对象：测试人员、开发人员、项目负责人、平台管理员

## 1. 平台入口与安全原则

测试平台覆盖需求、知识库、用例、计划、API 测试、UI 自动化、音视频专项、报告、缺陷、通知和审计。

- 本地测试入口：`http://localhost:5173`
- Docker 测试入口：`http://localhost`
- 后端健康检查：`http://localhost:8000/health`，Docker 环境为 `http://localhost/health`
- API 文档：`http://localhost:8000/docs`

平台不提供可公开复用的默认密码。账号由管理员创建，首次登录后应立即修改个人密码。

以下内容禁止写入 Git、需求、用例、截图或测试报告：

- 登录密码、JWT、API Token、Webhook 完整地址和 SMTP 授权码。
- AI API Key、蓝湖 Cookie/密码、数据库密码、VPN 配置和私钥。
- 体育测试或生产账号、支付凭据、推流密钥。

真实密钥只允许保存在未跟踪的 `.env`、CI/CD Secret 或平台“环境变量”的加密字段中。

## 2. 普通用户快速开始

### 2.1 登录与切换项目

1. 打开平台入口。
2. 输入管理员分配的用户名和密码。
3. 首次登录按提示修改密码。
4. 在页面顶部选择当前项目；后续需求、用例、环境和报告都按当前项目隔离。

### 2.2 新项目接入

1. 进入“项目管理”，创建项目编码、名称和描述。
2. 在“系统管理”给项目成员分配测试、开发、项目负责人或自定义角色。
3. 进入“环境管理”，创建 `dev`、`test`、`staging` 或 `prod` 环境。
4. 填写项目基础地址；Token、账号、密码勾选“加密存储”。
5. 使用变量名引用数据，例如 `${ACCESS_TOKEN}`、`${USER_ID}`、`${GATEWAY_HOST}`。

环境建议至少包含：

| 变量 | 用途 | 是否加密 |
| --- | --- | --- |
| `BASE_URL` | 项目网关地址 | 否 |
| `GATEWAY_HOST` | 反向代理 Host | 否 |
| `ACCESS_TOKEN` | 登录鉴权 Token | 是 |
| `USER_ID` | 当前测试用户 ID | 否 |
| `TEST_PHONE` | 测试账号 | 是 |
| `TEST_PASSWORD` | 测试密码 | 是 |

## 3. 需求、知识库与 RAG

### 3.1 导入需求

1. 进入“需求管理”。
2. 上传 Markdown、Word、Excel，或粘贴已授权的蓝湖地址。
3. 蓝湖证据采集先预览页数和选项，再发起任务。
4. 等待页面证据、OCR、Word/JSON 导出和需求导入完成。
5. 检查失败页和质量结果；失败任务使用“重试”，不要重复创建相同任务。

### 3.2 使用知识库

1. 进入“知识中心”，确认知识源状态正常。
2. 对新增知识源执行切片和向量回填。
3. 使用关键词、向量或混合检索。
4. 检查“向量健康”：可用切片数应等于总切片数。
5. Wiki 和差异结果属于待评审内容，评审后再导入正式用例或需求库。

RAG 使用本地 `BAAI/bge-small-zh-v1.5` 向量模型。模型缓存属于运行数据，不提交 Git；部署时通过持久化目录保存。

## 4. API 测试

### 4.1 导入接口资产

1. 进入“接口测试 → 接口资产”。
2. 选择 OpenAPI URL、JSON/YAML 文件或文本导入。
3. 先点击“预览”，核对服务名、新增数量和重复数量。
4. 确认后导入；平台会生成候选 API 用例。

候选用例不代表已获得执行授权。批量执行前必须排除：

- DELETE 和逻辑删除。
- 支付、扣款、退款、转账、提现。
- 发布、上下架、封禁、注销、发送短信、推流。
- 批量写入或无法隔离测试数据的接口。

### 4.2 快速调试

1. 选择环境。
2. 选择请求方法并填写相对 URL。
3. Header 使用变量，例如：

```json
{
  "Host": "${GATEWAY_HOST}",
  "accesstoken": "${ACCESS_TOKEN}"
}
```

4. POST/PUT 请求填写 JSON Body。
5. 发送后同时检查 HTTP 状态、业务状态、响应字段和响应时间。

### 4.3 推荐的安全增改查链路

1. 查询一个唯一且不存在的名称，确认返回空结果。
2. 使用唯一前缀新增低影响测试数据。
3. 查询并保存新记录 ID。
4. 按 ID 修改可逆字段。
5. 再次查询，核对 ID 不变且字段已更新。
6. 没有删除授权时保留记录并在报告中标注。

## 5. UI 自动化

1. 进入“UI 自动化”，创建脚本并选择运行环境。
2. `BASE_URL` 从所选环境注入，账号密码从加密变量或运行环境注入。
3. 点击执行后查看实时状态、步骤、控制台、截图、视频、Trace 和 HTML 报告。
4. 每次运行使用独立产物目录，避免并发任务互相覆盖。
5. 生产环境默认只做页面访问、元素存在和授权登录；任意点击或写操作必须单独评审。

## 6. 音视频专项

进入“专项测试”后可记录：

- 视频延迟、连麦延迟。
- 音画同步偏差。
- 视频帧率。
- 首帧加载耗时。

每个任务填写场景、采集方法、网络、设备、阈值、比较方向和样本。平台计算平均值、中位数、最小值、最大值、标准差和 P95，并判断是否达标。

帧率可以使用真实视频执行 `ffprobe`；直播延迟、音画同步和连麦需要已授权的直播间、推流端或采集设备。平台不会用随机数伪造真实测量结论。

## 7. 用例、计划、报告与缺陷

1. 在“用例库”创建或导入用例，完善优先级、前置条件、步骤和预期结果。
2. 在“测试计划”选择用例和环境，发起执行。
3. 执行结果使用通过、失败、跳过或阻塞，并填写实际结果。
4. 在“报告中心”生成快照报告，核对通过率和质量门禁。
5. 失败项可创建缺陷，按确认、修复、待验证、关闭等状态推进。
6. 使用“质量追溯”检查需求、用例、执行、报告和缺陷的关联完整性。

## 8. 钉钉 Webhook 与 SMTP 通知

### 8.1 钉钉

1. 在钉钉项目群添加“自定义机器人”。
2. 推荐安全模式使用自定义关键词 `测试`。
3. 复制完整 Webhook 地址，不要写入文档或 Git。
4. 在平台“通知中心”新增 Webhook，提供商选择“钉钉”。
5. 订阅 `task_started`、`task_finished`、`test_result`。
6. 先发送测试消息，成功后再启用通道。

当前平台支持钉钉 Markdown 消息。若机器人必须使用“加签”模式，需要另外配置 Secret 和签名能力。

### 8.2 SMTP

SMTP 参数只写入部署 `.env`：

```dotenv
SMTP_HOST=
SMTP_PORT=587
SMTP_USER=
SMTP_PASSWORD=
SMTP_FROM=
```

使用应用专用密码或授权码，不使用个人邮箱登录密码。

## 9. 管理员安全部署

### 9.1 Docker 部署

```powershell
cd F:\CamelTv\test-platform-v2\deploy
Copy-Item .env.example .env
```

在 `.env` 中至少设置：

- `SECRET_KEY`
- `ADMIN_PASSWORD`
- `TESTER_PASSWORD`
- `POSTGRES_PASSWORD`

随机值可在本机生成，不要把生成结果复制到聊天、工单或 Git：

```powershell
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

启动并验证：

```powershell
docker compose up -d --build
docker compose ps
Invoke-WebRequest http://localhost/health -UseBasicParsing
```

### 9.2 数据库迁移与备份

容器启动会运行 `alembic upgrade head`。升级前必须备份 SQLite 数据库或 PostgreSQL：

```powershell
Copy-Item .\backend\data\platform.db ".\backups\platform-$(Get-Date -Format yyyyMMdd-HHmmss).db"
```

不要使用 `docker compose down -v` 做日常重启，该命令会删除持久化数据。

### 9.3 密钥轮换

发生下列情况时立即轮换：密钥进入 Git、截图、日志、聊天、测试报告，成员离职，机器人地址外泄，或第三方平台提示异常调用。

轮换顺序：撤销旧值 → 生成新值 → 更新 `.env`/Secret 管理 → 重启服务 → 验证登录/通知/AI → 清理旧会话。

## 10. 升级、回滚与验收

升级：

1. 备份数据库和本地 `.env`。
2. 拉取指定 `master` 提交。
3. 运行后端测试、前端测试和生产构建。
4. 执行数据库迁移并重启服务。
5. 从 Web 验证登录、项目、环境、API 测试、RAG 和通知。

回滚：

1. 停止新版本服务。
2. 切回已验收的 Git 提交。
3. 仅在迁移不可向前兼容时恢复升级前数据库备份。
4. 启动并重新执行健康检查和 Web 冒烟。

发布验收至少包含：

- `/health` 返回 200。
- 登录页不预填账号密码。
- DEV 用户可进入授权模块。
- API 资产页能加载、导入和快速调试。
- RAG 关键词/向量/混合检索有结果。
- 音视频专项能新增、查询和修改真实样本。
- UI 任务有独立产物目录。
- 钉钉/SMTP 测试通知成功。
- Git 敏感信息扫描无真实凭据。

## 11. 常见问题

| 问题 | 处理方式 |
| --- | --- |
| 登录失败 | 确认账号状态、项目角色和管理员分配的密码；不要尝试仓库历史中的旧密码 |
| API 页面空白或崩溃 | 清理浏览器缓存并确认前后端版本一致；检查 `/health` 和浏览器控制台 |
| 内网 API 不通 | 确认 VPN、路由、DNS 和网关 Host；不要修改全局 hosts 影响其他项目 |
| Token 无效 | 重新通过目标 Web 登录并更新平台加密变量；不同后台可能使用独立 Token |
| RAG 无向量 | 检查模型缓存和向量健康，重新执行待处理切片回填 |
| UI 任务无报告 | 检查 Playwright 运行目录权限、浏览器依赖和任务产物路径 |
| 通知失败 | 检查 Webhook/SMTP 状态、订阅事件、关键词/加签方式和发送日志 |

需要新增外部系统授权时，应只提供最小权限的测试环境地址和测试账号，并明确允许的读写范围。
