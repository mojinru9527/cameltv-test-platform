# batch-23 PRD 摘要 — 接口测试模块优化

> 日期：2026-07-20 | 产品部门 | 7 项用户体验和功能增强

## 问题陈述

当前测试平台「接口测试」模块存在 7 个用户体验和功能不足的问题：

1. **列表展开混乱**：接口资产列表中服务名、模块名未按用户预期组织，层级关系不清晰
2. **Tab 与模块关系错位**：服务名 Tab 已存在，但选中 Tab 后仍然展示所有服务的数据混合展示，未做服务级数据隔离
3. **三层层级不清晰**：服务 → 模块 → 接口的三级层次关系未准确反映在 UI 上
4. **快速调试布局不佳**：响应结果未在页面下方单独展示，用户体验不连贯
5. **用例组织碎片化**：接口用例列表中用例散列，未按所属接口分组聚合
6. **地址拼接体验差**：服务器地址与服务名/模块名/接口名未做拆分，切换环境时体验不直观（部分已实现需加固）
7. **用例生成覆盖不足**：自动生成用例数量少，多参数场景下缺乏类型覆盖和边界值（空值/null）覆盖

## 为什么用户关心

- **导航效率**：当成百上千个接口资产导入后，不收起默认展开导致页面极其冗长，无法快速定位目标接口
- **数据隔离**：不同微服务（如用户服务、订单服务）的接口混在一起，测试人员需要按服务视角逐个审查
- **调试效率**：响应结果在配置表单中间，需要反复滚动页面查看请求和响应，打断调试心流
- **用例管理**：一个接口生成 5 条用例后，这些用例散落在列表各处（按标题排序），无法按接口维度审视覆盖率
- **测试质量**：自动生成的用例太少（仅 4 种模板、每个字段 1-2 条），多参数组合、类型覆盖、空值校验等边界条件缺失，导致测试覆盖不足

## 成功指标

| 指标 | 当前 | 目标 |
|------|------|------|
| 接口资产列表默认折叠 | 模块级折叠 | 选服务Tab后只显示该服务下模块（折叠），点击展开 |
| 服务Tab数据隔离 | 按service_id过滤API | 保持不变，确认正确 |
| 层级展开体验 | 模块→路径组→接口（三级） | 服务Tab → 模块（折叠） → 接口列表（二级+内联路径组） |
| 快速调试响应位置 | 请求卡片下方 | 保持在底部（已实现，确认无退化） |
| 用例按接口分组 | api_spec_ref 分组 | api_endpoint + 接口名称分组 |
| 地址拆分 | 已实现4字段 | 加固体验，确认环境切换联动正确 |
| 用例生成覆盖 | 4 templates, 10-20条/接口 | 6 templates, 30-60条/接口，每个参数3+用例 |

## 非目标

- 不新增接口测试模块以外的功能
- 不修改数据库 schema
- 不改变现有 API 契约（仅扩展前端展示和用例生成逻辑）
- 不碰 `lanhu-mcp/`、`deploy/`、`tests/` 模块
- 不引入新的第三方依赖

## 用户故事

### US-1: 接口资产 — 默认收起列表

**作为** 测试工程师
**我想** 进入接口资产页面时，服务名tab下只展示模块名（收起状态），点击模块才展示下面的接口
**以便** 快速浏览有哪些模块，而不是被大量接口列表淹没

**验收标准：**
- Given 接口资产页面加载完成，When 用户选中某个服务 Tab，Then 仅展示该服务下的模块名列表（全部折叠），不展示路径组/接口
- Given 模块名列表已展示，When 用户点击某个模块名（展开），Then 展开该模块下的路径组和接口列表
- Given 切换到另一个服务 Tab，When 新 Tab 加载完成，Then 自动收起前一个服务的展开状态，展示新服务的折叠模块列表

### US-2: 服务 Tab 数据隔离

**作为** 测试工程师
**我想** 点击不同的服务 Tab 时只看到该服务下的模块和接口
**以便** 按微服务维度审查接口资产

**验收标准：**
- Given 存在服务A和服务B两个Tab，When 选中服务A Tab，Then 列表仅展示服务A下的模块和接口
- Given 选中「全部服务」Tab，When 列表加载完成，Then 展示所有服务下的所有模块（按服务分组显示）

### US-3: 层级对应关系

**作为** 测试工程师
**我想** 看到清晰的三层关系：服务 → 模块 → 接口
**以便** 理解每个接口在系统中的位置归属

**验收标准：**
- Given 接口资产页面，When 选中服务Tab，Then 先显示模块名集合
- Given 展开某个模块，When 模块下的接口展示，Then 接口按路径分组（路径组头）内联显示在模块下方
- Given 接口列表，When 查看接口行，Then 每行显示方法、路径、描述、备注

### US-4: 快速调试响应结果底部展示

**作为** 测试工程师
**我想** 在快速调试页面发送请求后，响应结果展示在整个配置表单的底部
**以便** 不需要在配置区中间来回滚动查看结果

**验收标准：**
- Given 快速调试页面，When 页面加载，Then 响应结果区域在页面最底部
- Given 发送请求后，When 响应返回，Then 页面自动滚动到响应结果区域

### US-5: 接口用例按接口集合展示

**作为** 测试工程师
**我想** 接口用例列表中，用例按所在接口聚合展示
**以便** 快速了解每个接口有哪些用例覆盖

**验收标准：**
- Given 接口用例列表，When 加载完成，Then 用例按接口路径分组，显示为"接口C（5条用例）→ C1,C2,C3,C4,C5"
- Given 分组已折叠，When 点击分组展开，Then 展示该接口下所有用例
- Given 接口A下生成5条用例，When 查看接口用例列表，Then 接口A显示为一个分组，下面有5条用例

### US-6: 服务器地址拆分与环境联动

**作为** 测试工程师
**我想** 在快速调试中分别填写服务器地址和服务名/模块名/接口名，切换环境时服务器地址自动变化
**以便** 在不同环境间快速切换调试

**验收标准：**
- Given 快速调试页面，When 首次加载，Then 默认选中「测试5」环境，服务器地址自动填充
- Given 切换环境从「测试5」到「测试3」，When 选中新环境，Then 服务器地址自动变更为测试3的地址
- Given 填写了服务器地址+服务名+模块名+接口名，When 点击发送，Then 请求URL按 baseUrl/serviceName/modulePath/endpointPath 拼接

### US-7: 用例生成增强覆盖率

**作为** 测试工程师
**我想** 自动生成接口用例时覆盖更多参数类型和边界值
**以便** 减少手动补写用例的工作量

**验收标准：**
- Given 接口有3个请求参数（string, integer, boolean），When 生成用例，Then 每个参数至少生成3条用例（正常值+类型错误+空值/null）
- Given 接口参数有枚举值，When 生成用例，Then 生成合法枚举+非法枚举至少2条
- Given 接口参数有 string 类型，When 生成用例，Then 覆盖空字符串、超长、特殊字符、SQL注入
- Given 接口参数有 format（email/date/uri），When 生成用例，Then 生成格式错误用例
- Given 一个接口有5个参数，When 点击生成用例，Then 生成至少30条用例覆盖各类场景
