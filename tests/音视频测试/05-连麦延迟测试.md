# 05-连麦延迟测试

> **模块说明**: 本模块用于测试直播连麦场景下的端到端延迟，包括主播与观众连麦、主播与主播连麦两种场景。连麦延迟通常比普通直播延迟更敏感，因为涉及双向实时交互。
> **前置条件**: 已完成 [01-环境安装指南](01-环境安装指南.md) 和 [02-测试素材准备](02-测试素材准备.md)
> **测试工具**: OBS Studio、Bandicam（2台设备各装）、opencv.py、analysis.py

---

## 1. 测试原理

```
场景A：主播-观众连麦
[主播A推流] ←→ [直播服务器] ←→ [观众B连麦上麦]
     ↓                                ↓
  录制A画面                       录制B画面
     ↓                                ↓
  对比两端时间戳差异 = 连麦双向延迟
```

```
场景B：主播-主播连麦(PK)
[主播A推流] ←→ [直播服务器] ←→ [主播B推流]
     ↓                                ↓
  录制A画面                       录制B画面
     ↓                                ↓
  分析A→B延迟 和 B→A延迟
```

## 3. 场景A：主播-观众连麦

### 3.1 场景描述

主播A用PC推流播放时间戳测试视频，观众B在手机端进入直播间并发起连麦，A接受后B上麦。同时录制两端画面分析延迟。

### 3.2 执行步骤

| 步骤 | 操作 | 详细说明 | 完成 |
|------|------|----------|------|
| 1 | 准备测试环境 | 主播A PC和观众B手机在同一网络下，确保网络稳定 | ☐ |
| 2 | 主播A开始推流 | OBS媒体源→选择时间戳测试视频→开始推流 | ☐ |
| 3 | 观众B进入直播间 | 手机APP→搜索直播间→进入 | ☐ |
| 4 | 等待推流稳定 | 约10秒，确认B能看到A的画面 | ☐ |
| 5 | **同时开始录制** | 手机2录制主播A的PC屏幕 + 观众B手机开启Bandicam录制自身屏幕 | ☐ |
| 6 | 观众B发起连麦 | APP内点击"连麦"按钮→等待主播A接受 | ☐ |
| 7 | 主播A接受连麦 | 确认连麦成功，B的声音/画面出现在直播间 | ☐ |
| 8 | 保持连麦状态 | 持续录制`[____]`秒（建议120秒） | ☐ |
| 9 | 结束录制 | 两端同时停止录制 | ☐ |
| 10 | 保存文件 | A端: `comicast_host_[场景编号].mp4` / B端: `comicast_viewer_[场景编号].mp4` | ☐ |

### 3.3 关键观察点

在连麦过程中注意观察（录制结束后立刻记录）：

| 观察点 | 说明 | 记录 |

| 连麦建立耗时 | 从B点击连麦到A接受后画面出现 | `[____]`秒 |
| 连麦画面延迟 | B端看到A画面的延迟 | 主观评价: `[______]` |
| 连麦音频延迟 | B说话到A听到的时间差 | 主观评价: `[______]` |
| 音画同步 | 连麦时声音和嘴型是否同步 | ☐ 同步 ☐ 轻微不同步 ☐ 严重不同步 |

## 6. 常见问题

| 问题 | 可能原因 | 解决方法 |
|------|----------|----------|
| 连麦失败 | 网络NAT穿透失败 | 检查两端网络类型，确保非对称NAT |
| 连麦建立很慢(>5秒) | 信令服务器延迟 | 检查信令服务器状态，换时间段重试 |
| 连麦时画面卡顿 | 上行带宽不足 | 降低推流码率，检查上传速度 |
| 音频回声 | 两端距离太近 | 双方使用耳机，拉开物理距离 |
| 双向延迟差异大 | 两端网络条件不对等 | 分别检查两端的网络状况 |
