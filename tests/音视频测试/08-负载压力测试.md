# 08-负载压力测试

> **模块说明**: 本模块用于测试直播服务在多观众同时观看时的承载能力，以及不同观众端的延迟/流畅度差异。通过多台设备同时进入同一直播间来模拟负载场景。
> **前置条件**: 已完成 [01-环境安装指南](01-环境安装指南.md)、[02-测试素材准备](02-测试素材准备.md)、[03-视频延迟测试](03-视频延迟测试.md)
> **测试工具**: OBS Studio、多台测试手机（3-5台）、Bandicam、opencv.py、analysis.py

## 3. 基础负载测试

### 3.1 执行步骤

| 步骤 | 操作 | 详细说明 | 完成 |
|------|------|----------|------|
| 1 | 所有手机安装Bandicam（或使用系统自带录屏） | 确保每台手机都有足够的存储空间（≥2GB） | ☐ |
| 2 | OBS开始推流 | 使用时间戳测试视频，循环播放 | ☐ |
| 3 | 等待推流稳定 | 约10秒 | ☐ |
| 4 | **同时操作** | 所有手机同时打开APP→进入同一个直播间 | ☐ |
| 5 | 确认所有手机画面正常 | 每台手机确认能正常看到直播画面 | ☐ |
| 6 | **同时开始录制** | 所有手机同时开始录屏（倒计时3-2-1同步） | ☐ |
| 7 | 保持录制 | 录制`[____]`秒（建议120秒） | ☐ |
| 8 | **同时停止录制** | 所有手机同时停止录屏 | ☐ |
| 9 | 保存文件 | 格式: `loadtest_[手机编号]_[分辨率].mp4` | ☐ |
| 10 | 停止推流 | OBS停止推流 | ☐ |

### 3.2 同步技巧

由于多台设备很难精确同步开始录制，使用以下方法：

**方法：声控同步**
1. 在开始录制后，主播端播放一声响亮的提示音（拍手/敲击）
2. 数据分析时以这声提示音作为时间零点对齐各设备的日志

### 3.3 数据处理

对每台手机的录制文件分别执行：

```cmd
python C:\直播测试脚本\opencv.py "C:\直播测试结果\[测试文件夹]\loadtest_手机1_xxx.mp4"
python C:\直播测试脚本\analysis.py "C:\直播测试结果\[测试文件夹]\loadtest_手机1_xxx.log"

python C:\直播测试脚本\opencv.py "C:\直播测试结果\[测试文件夹]\loadtest_手机2_xxx.mp4"
python C:\直播测试脚本\analysis.py "C:\直播测试结果\[测试文件夹]\loadtest_手机2_xxx.log"

...依此类推
```

## 8. 常见问题

| 问题 | 可能原因 | 解决方法 |
|------|----------|----------|
| 某台设备延迟明显高于其他 | 该设备网络状况差/性能不足 | 单独检查该设备的网络和性能 |
| 观众数增多后画质下降 | 服务器自动降低码率(ABR) | 检查服务器是否有观众数相关的码率策略 |
| 多台设备结果难以同步 | 录制开始时间不一致 | 使用声控同步法（见3.2节） |
| 存储空间不够 | 多台设备同时录制 | 提前清理各设备存储，预估每台需要≥2GB |
