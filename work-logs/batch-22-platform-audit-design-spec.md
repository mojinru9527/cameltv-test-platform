# Batch 22 — 测试平台全面审查 Design Spec

> **Design (🎨)** | Date: 2026-07-19 | Status: Draft

---

## 0. 设计基线确认

- **技术栈**: shadcn/ui (Radix + Tailwind + CVA)，非 Ant Design
- **Token 体系**: CSS 变量 via `globals.css`——5 个 `data-theme-id` 主题 (crystal/xlab/column/clay/liquid) + `.dark` 变体
- **组件体系**: `components/ui/*` (34 个 shadcn 生成) + 9 个自定义组件 (AsyncState/DataTable/EmptyState/ErrorBoundary/PageHeader/SearchInput/StatCard/Pagination/CommandPalette) + hooks
- **设计参照**: 以知识中心 SearchTab 为样式基线（`cameltv-ui-conventions` skill §样式基线）

## 1. 「小白测试」用户旅程——体验断点走查

按 Product PRD 四阶段逐页走查：

### 阶段 A：登录 → 首页

| 页面 | 状态 | 发现 |
|------|------|------|
| `/login` | ✅ 功能可用 | 无品牌 LOGO/标题，纯表单；Loading 仅在按钮（`loading ? Loader2 : "登 录"`），表单级无骨架；无 ErrorState（仅 toast） |
| `/workbench` | ✅ 生产可用 | 使用 AsyncState + useApi，Loading/Error/Empty 三态齐全；图表响应式 OK |

### 阶段 B：需求提交 → AI 生成 → 审查

| 页面 | 状态 | 发现 |
|------|------|------|
| `/requirement` 列表 | ⚠️ | 上传入口在二级操作区（需要点「上传」才看到），「快速创建」不存在 |
| `AiResultModal` | ❌ 核心 UX 缺陷 | ① 一次性弹窗——关闭后无法返回审查 ② 全选/全不选，无按优先级/类型筛选 ③ 无 diff 对比（与上次生成的区别） ④ 无 approve/reject 操作记录 |

### 阶段 C：测试计划 → 执行 ❌（不存在）

| 页面 | 状态 | 发现 |
|------|------|------|
| `/testplan/:id` | ⚠️ | 执行是逐条手工的——每条用例一个「执行」按钮 → 弹窗选 pass/fail/skip/block → 填备注。无「一键全部执行」按钮，无执行进度条，无批量执行结果视图 |
| 执行结果展示 | ❌ | 执行完成后无智能分诊——所有失败平铺，无「AI 认为这是真 bug/环境抖动/用例缺陷」的分类 |

### 阶段 D：查看报告 → 提缺陷

| 页面 | 状态 | 发现 |
|------|------|------|
| `/report/:id` | ⚠️ | 有 JSON 快照 + 统计 + 趋势图。但无数「从失败用例一键提缺陷」通道——需要切换到 defect 页面手工创建 |

## 2. 全局 UI 一致性走查

与 `cameltv-ui-conventions` skill 的 8 类 Red Flags 逐条比对，检视所有 13 个业务页面：

### 🟢 已对齐（无问题）

- **样式基线**: 工作台使用 `AsyncState`+`useApi`——正确的四态范式
- **组件复用**: `PageHeader`/`StatCard`/`DataTable`/`SearchInput`/`Pagination` 在各列表页一致使用
- **焦点**: globals.css `*:focus-visible` ring 全局覆盖
- **反馈**: sonner toast 统一

### 🟡 P2 问题（影响一致性和体验）

| # | 问题 | 影响页面 | 修法 |
|---|------|---------|------|
| D1 | 登录页无品牌标识 | `/login` | 加平台 LOGO + 标题 + 副标题（"CamelTv 测试平台"） |
| D2 | 登录页缺三态 | `/login` | Loading 时整表单 disabled+skeleton；Error 时内联错误提示（非仅 toast）；无 Empty 场景 |
| D3 | 上传入口深藏 | `/requirement` | 「快速创建」作为页面主 CTA，而非藏在操作区 |
| D4 | `AiResultModal` 一次性 | `/requirement` | 改为持久化审查页（见 PM plan Task 2a） |
| D5 | 无执行进度可视化 | `/testplan/:id` | 批量执行时需进度条 + 实时状态流（SSE/polling） |
| D6 | 无「快速操作」入口 | 全局 | 工作台或顶栏加「提交需求」「快速执行」快捷入口——降低小白导航成本 |
| D7 | 空态引导不足 | 多页面 | 空态不只是「暂无数据」，应引导下一步（"上传第一份需求文档" + 按钮） |
| D8 | 响应式：Tab 过多 | `/knowledge`（9 Tab）| 平板 768px 以下溢出；但 13 个主页面 Tab ≤ 4，无此问题 |

### 🔴 P1 问题（影响小白可用性）

| # | 问题 | 影响页面 | 修法 |
|---|------|---------|------|
| D9 | **无新用户引导** | 全部 | 首次登录无 onboarding wizard、无 tooltip 引导、无「第一步做什么」——纯靠读 `onboarding.md` 文档。需一个 3 步引导蒙层 |
| D10 | **演示态页面未标注** | `/apitest` `/uitest` `/special` | 如果仍为演示态，页面顶部需醒目标注「演示数据，非真实测试」；如果已做真，更新标注并移除「演示态」标签 |
| D11 | **执行操作碎片化** | `/testplan/:id` | 逐条执行 50 条用例 = 点 50 次按钮——即使不做自动执行，也需要「批量执行」按钮（逐条自动跑 + 进度条） |

## 3. 无障碍审计（快扫）

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 表单 label 关联 | ✅ | 登录页 Input 有 Label（通过 shadcn Input 组件） |
| 焦点可见 | ✅ | globals.css 全局 `*:focus-visible` ring |
| 图标按钮 aria-label | ⚠️ | 未全覆盖——需逐页用 axe DevTools 跑一次（不在本次手工走查范围） |
| 颜色对比度 | ✅ | Text `muted-foreground` 已通过 WCAG AA（4.6:1 on white），主题 Token 已含 dark 变体 |

```
┌──────────────────────────────────────────────────────┐
│ ← 返回需求列表    需求审查 · {需求标题}    [全部导入] │

│ [新增]       │       3. 点击登录 4. 验证跳转首页     │
```

## 5. 设计签核

**结论：有条件通过（Conditional Pass）**

整体架构（shadcn/ui + Tailwind + AsyncState）基线扎实，工作台/列表页范式一致。但存在 **3 项 P1 阻断**（新用户引导、演示态标注、执行操作碎片化）和 **8 项 P2 优化**（见上表），需在 Slice 2 解决。

**要求**：
- D9（新用户引导）→ Slice 2 与审查队列一起交付
- D10（演示态标注）→ Slice 0 事实核查后立即跟新
- D11（批量执行 UX）→ Slice 1 执行通道的一部分
