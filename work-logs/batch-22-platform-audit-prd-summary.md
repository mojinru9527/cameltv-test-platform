## 1. 问题陈述：测试平台的「最后一公里」断裂

### 愿景

> 一个小白测试人员，只需要做三件事：**提交需求文档 → 审查 AI 生成的用例 → 审查 AI 完成的测试结果**。其他一切由平台自动完成。

### 现实

```
需求提交 ✅ → AI生成用例 ✅ → 用例库 ✅ → 测试计划 ✅
                                          ↓
              AI自动执行 ❌  ← ← ← ← ← ← ← 【断裂点】
                                          ↓
          小白审查结果 ⚠️  ← 根本没有走到这一步
```

**V2.2–V2.6 的 36 个交付项全部完成**，管理闭环（需求→用例→计划→报告/缺陷）已生产可用。但**从「AI 生成功能用例」到「AI 自动执行这些用例」之间的桥梁不存在**：

1. **功能用例只能手工逐条执行**（`POST /test_plan/{id}/cases/{pcase_id}/execute`），无法批量、无法自动化
2. **API 测试引擎**（httpx 真实执行）虽已从演示态升级为真实执行，但**不能消费 AI 生成的接口用例**——AI 生成的 `api_cases[]` 导入用例库后就停在文本层面，没有"一键转可执行请求"的通道
3. **UI 自动化引擎**（Playwright 子进程）需要手写 `.spec.ts` 规格文件，**不能从功能用例文本自动生成 Playwright 脚本**
4. **音视频专项**仍是 `random.uniform` 模拟（backlog 宣称已做真，但 CLAUDE.md 标注 🧪 演示态，现状 PRD 也确认随机数——**文档自相矛盾，需核查代码**）
5. **审查体验缺失**：AI 生成 50 条用例后，小白没有一个「批量预览 + 逐条 approve/reject + 一键导入」的流畅界面；执行完成后也没有「AI 帮你判断哪些失败值得关注」的智能分诊

### RICE 评分

| 维度 | 评分 | 理由 |
|------|------|------|
| Reach（覆盖用户） | 3 | 所有测试人员 + 产品/开发自测 |
| Impact（影响程度） | 3 | 从「手工为主」变「AI 自动化为主」，效率量变到质变 |
| Confidence（信心） | 80% | AI 生成用例已验证可行；自动化执行有 API 引擎基础 |
| Effort（人月） | 3-4 | 需要：用例→脚本编译器 + 批量执行编排 + 智能分诊 UI |

## 4. 按「小白测试」用户旅程逐段诊断

### 阶段 A：小白提交需求文档

**当前状态**：✅ 可用。上传 MD/Word/Excel/蓝湖链接 → AI 两段式生成（需求分析 + 用例生成），含「反向评审」指出需求自身问题。

**剩余差距**：
- 上传入口藏在「需求管理」页的二级操作里，新用户第一眼找不到
- 蓝湖链接提取路径硬编码，换部署即失效
- 生成失败时只有通用 toast，无「重试/换模型/手动补充」的降级引导
- 无模板：「我不知道怎么写需求文档」——可提供"一句话需求"输入 + AI 帮你展开为结构化需求

### 阶段 B：AI 生成用例 → 小白审查

**当前状态**：⚠️ 功能存在但体验割裂。`AiResultModal` 展示生成结果，勾选导入。

**剩余差距**：
- 审查是「一次性弹窗」，关了就没机会重新审——应该有个「审查队列」持久化
- 无 diff 对比：AI 改进了什么？与上次生成有什么不同？
- 无批量操作：50 条用例只能逐条勾选或全选，不能按「只看 P0/只看接口/只看修改过的」筛选
- 审查反馈不闭环：小白 reject 一条用例，AI 学不到——下一次生成还是会犯同样的错误

### 阶段 C：AI 自动执行 ❌（核心断裂）

**当前状态**：❌ 不存在。这是小白愿景最大的障碍。

功能用例（`case_type=manual`）执行完全靠人：打开页面 → 按步骤操作 → 填入结果。没有 Playwright 脚本生成、没有步骤解析、没有自动化执行引擎。

API 用例（`case_type=api`）虽有 httpx 执行引擎，但 AI 生成的 `api_cases[]` 停留在文本——需要手工把 `api_method/endpoint/headers/body` 转成可执行请求，没有「一键执行」。

**需要的核心能力**：
1. **用例→脚本编译器**：功能用例的 `steps` JSON → Playwright spec；API 用例的字段 → httpx request
2. **批量执行编排**：提交需求后一键「生成+执行」，自动创建计划→加入用例→逐条执行→生成报告
3. **智能分诊**：执行完成后 AI 预分析"哪些失败是真 bug、哪些是环境抖动、哪些是用例写错了"，小白只看 AI 标出来的问题

### 阶段 D：小白审查测试结果

**当前状态**：⚠️ 报告中心有 JSON 快照 + 趋势图，但为「管理者视角」设计。

**剩余差距**：
- 无「测试结论摘要」——AI 帮你读报告，输出一段话："本次测试共 45 条用例，通过 38 条，失败 7 条。其中 3 条疑似环境问题，2 条为已知缺陷 #42，2 条需人工确认"
- 无「一键提缺陷」——从失败用例到创建缺陷，中间要切换页面、手工填字段
- 无「回归对比」——这次跑的和上次跑的，哪些用例从 pass 变 fail 了？

## 5. 文档体系自相矛盾（本次审查发现的元问题）

| 文档 | 声称 | 矛盾 |
|------|------|------|
| `CLAUDE.md` 模块表 | API 测试 🟡、UI 自动化 🟡、音视频 🧪 | 与 backlog 声称的 V2.4 "三引擎做真" 矛盾 |
| `现状功能PRD.md` | API 测试「纯前端 fetch」、UI/AV「随机数」 | 同上——需要代码级验证哪个是真的 |
| `改进任务backlog.md` | V2.4-V2.6 36/36 100% 完成 | 但 CLAUDE.md 和现状 PRD 未更新 |
| `代码审查PRD.md` | "0 自动化测试、文档与代码脱节"（2026-06-22） | 一个半月后，同样的问题依然存在 |

**结论**：在做任何功能改进之前，必须先做一次**代码级事实核查**——确认 API/UI/AV 三个引擎到底是不是真的、文档更新滞后到底漏了多少。

## 6. 推荐优先级

| 优先级 | 改进项 | 理由 |
|--------|--------|------|
| **P0** | 代码级事实核查：API/UI/AV 引擎真实状态 + 文档同步 | 连现状都不清楚就别谈改进 |
| **P0** | 功能用例自动执行通道（用例→Playwright 脚本 + 批量编排） | 小白愿景的核心断裂点 |
| **P0** | API 用例一键执行（AI 生成 → 可执行请求，零手工） | 接口测试是最大 ROI（httpx 引擎已有基础） |
| **P1** | 审查体验重构（审查队列 + 智能分诊 + 批量操作 + 一键提缺陷） | 小白审查效率从"逐条翻阅"变"只看 AI 标的" |
| **P1** | 需求输入简化（"一句话需求" + 模板 + 蓝湖路径配置化） | 降低小白第一步门槛 |
| **P2** | 审查反馈闭环（AI 学习 reject 模式） | 长期提升生成质量 |
| **P2** | 演示态全部做真并标注 | 信任问题——别让用户发现"这个指标是假的" |

## 7. 技术考量

- **用例→脚本编译器**：可以利用 LLM 完成——functional case steps → Playwright test code，质量靠 sandbox 执行验证
- **批量执行编排**：需要 TaskQueue（当前已有 APScheduler，可扩展为通用任务队列）
- **智能分诊**：将执行结果 + 历史数据喂给 LLM，输出分类（真 bug / 环境抖动 / 用例缺陷）——和 AI 生成用例是同一个 LLM 管道
- **依赖**：上述能力依赖 API/UI 引擎做真（如果 backlog 的 V2.4 交付属实则已有基础；如果文档准确则需要从零构建）
