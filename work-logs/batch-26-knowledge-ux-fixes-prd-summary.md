# Batch 26-2 — 知识中心 UX 修复 PRD Summary
> **Product (🟦)** | Date: 2026-07-21 | Status: Draft

## 1. 问题陈述

用户完成知识中心全功能走查后，发现 7 类 UX 问题。核心痛点：**页面能渲染数据但不可交互**——卡片/列表项点击无响应、弹窗内容不可读、图谱混淆两个知识域、缺少批量操作和溯源信息。

### 证据
- 用户逐功能走查全模块（15 个 API 全部通过，但前端交互链断裂）
- 项目知识 Tab：展示 12 批次 21 条知识源，但条目是纯 `<div>`，无 onClick
- 平台研发 Tab：展示 5 分区 101 条知识源，同上
- 所有弹窗的 Select 下拉错位，内容显示过小
- AI 审核台缺批量采纳/驳回

### 为什么用户关心
知识中心是平台核心模块（已投入 M0-M6 十张表 + 26 个 API），但如果前端交互链断裂，对用户而言就等于「不可用」。每个问题都直接阻断用户完成日常任务：查看知识详情、检索历史数据、审核 AI 产物。

## 4. 用户故事 + 验收标准

### P1-1：项目知识 Tab — 点击知识源查看详情
- **As a** 测试人员, **I want** 在项目知识 Tab 点击知识源条目查看完整内容, **so that** 我能了解每条知识的原始内容和 AI 加工结果。
- **Given** 用户在项目知识 Tab 看到批次卡片和知识源条目 / **When** 点击某条知识源 / **Then** 弹出详情弹窗，展示原始内容 + AI 切片 + 元数据（复用 SourceListTab 已有弹窗组件）
- **Given** 弹窗已打开 / **When** 查看内容 / **Then** 弹窗宽度 ≥ 屏幕 90%，高度 ≥ 90vh，所有文字清晰可读（text-sm 起步）

### P1-2：平台研发 Tab — 折叠分区 + 点击知识源查看详情
- **As a** 平台开发者, **I want** 分区默认折叠且点击展开，点击知识源条目查看详情, **so that** 5 个分区 101 条知识源不会一次性铺满屏幕。
- **Given** 用户在平台研发 Tab / **When** 首次加载 / **Then** 仅第一个分区（area 问题模式）展开，其余折叠
- **Given** 分区折叠 / **When** 点击分区标题 / **Then** 该分区展开/收起切换
- **Given** 分区展开 / **When** 点击某条知识源 / **Then** 弹出详情弹窗（复用 SourceListTab 弹窗）

### P1-3：概览 Tab 前移 + 默认显示
- **As a** 项目管理者, **I want** 进入知识中心首先看到全局仪表盘, **so that** 快速了解知识资产整体状况。
- **Given** 用户点击左侧导航「知识中心」 / **When** 页面加载 / **Then** 默认显示概览 Tab（而非项目知识）
- **Given** Tab 栏 / **When** 查看 Tab 顺序 / **Then** 概览在第一位

### P2-1：检索模块提升到页面顶部
- **As a** 所有用户, **I want** 在知识中心页面顶部随时使用搜索栏, **so that** 不用先切换到「检索」Tab 再搜索。
- **Given** 用户在知识中心任意 Tab / **When** 查看页面顶部 / **Then** 搜索栏常驻显示（在 Tab 栏上方或 PageHeader 下方）
- **Given** 用户输入检索词 / **When** 执行搜索 / **Then** 检索全部状态的知识切片（active/deprecated/archived），不仅限审核通过的

### P2-2：图谱模块区分知识域
- **As a** 测试人员/开发者, **I want** 项目知识和平台研发的图谱分开查看, **so that** 不会把体育项目的 API 实体和平台研发的技术决策混在一起。
- **Given** 用户在图谱 Tab / **When** 查看 / **Then** 顶部有「项目知识」/「平台研发」切换开关（默认项目知识）
- **Given** 切换知识域 / **When** 重新加载图谱 / **Then** 节点和边仅来自所选知识域
- **注**：层级调整方案用户后续会提供详细需求，本次仅做域隔离

### P3-1：弹窗 Select 下拉错位修复
- **As a** 所有用户, **I want** 弹窗中的下拉菜单正确定位, **so that** 不会被截断或偏移。
- **Given** 弹窗中有 Select 组件 / **When** 点击打开下拉 / **Then** 下拉菜单紧贴触发器，不错位。根因：Radix Select 的 Portal 默认挂到 document.body，而 Dialog 也用了 Portal，z-index 叠加时 Select content 可能被 Dialog 的 stacking context 影响。修复：给 SelectContent 加 `z-50` 或使用 `position="popper"`。

### P3-2：所有弹窗内容放大 5 倍
- **As a** 所有用户, **I want** 弹窗中的文字和内容清晰可读, **so that** 不需要放大镜查看知识详情。
- **Given** 任意弹窗 / **When** 查看内容 / **Then** 内容区最小 text-sm（14px），代码/原始内容最小 text-sm，弹窗宽度 ≥ max-w-4xl，高度 ≥ 85vh

### P3-3：AI 审核台批量操作
- **As a** QA 工程师, **I want** 批量采纳/驳回/导入 AI 产物, **so that** 不用逐条点击。
- **Given** AI 审核台列表 / **When** 勾选多条待审核产物 / **Then** 出现「批量采纳」「批量驳回」按钮
- **Given** 勾选多条已采纳产物 / **Then** 出现「批量导入」按钮
- **注**：ArtifactReviewTab 已有 checkbox 列和批量导入，需扩展支持批量采纳/驳回

### P4-1：知识溯源字段增强
- **As a** 知识管理员, **I want** 每条知识都能追溯到来源项目、模块和数据出处, **so that** 能评估知识的权威性和适用场景。
- **Given** 查看知识源详情弹窗 / **When** 查看元数据区 / **Then** 清晰展示：所属项目名 → 所属模块 → 数据来源路径/URL
- **涉及**：KnowledgeSource 模型增加 `module_name` 字段，前端详情弹窗增加溯源信息行

## 5. 技术考量

| 依赖/风险 | 影响 | 缓解 |
|-----------|------|------|
| Select 弹窗错位是 Radix 已知问题 | 中 | position="popper" + 调整 z-index |
| 图谱域隔离需要后端 API 支持 knowledge_domain 筛选 | 中 | 后端已有 knowledge_domain 字段，需前端传参 |
| 检索搜索全状态切片需要后端 SearchService 去除 status 过滤 | 中 | 确认后端 searchKnowledge 的 WHERE 条件 |
| 批量采纳/驳回 API 需确认后端是否支持 | 低 | 现有 API 逐条操作，前端并发调用或新增批量端点 |
| 溯源字段 module_name 需数据库迁移 | 低 | Alembic 添加可空字段 |
