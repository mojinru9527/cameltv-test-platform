# 需求文档模块重新规划方案 v2 — 含版本差异检测

> Product Department Proposal | 2026-07-21

## 1. 现状问题

| # | 问题 | 影响 |
|---|------|------|
| 1 | 证据采集任务创建后不可见，关闭侧边栏就找不到 | 用户不知道任务跑没跑 |
| 2 | 任务不自动执行，需手动点刷新才能触发 worker | 流程断裂，用户困惑 |
| 3 | 任务状态/进度/历史不可查 | 无法追溯，重复提交 |
| 4 | OCR 文本不足以反映图片为主的蓝湖原型 | AI 拆分质量差 |
| 5 | 功能拆分结果不能体现完整需求 | 用户看不到完整文档 |
| 6 | 同一蓝湖链接多个版本时，无法区分变动 | **核心痛点** |

## 2. 核心洞察

蓝湖设计稿的特点：
- **以图片为主，文字为辅**：原型截图是主要信息载体，OCR 文字只是补充
- **同一链接多版本迭代**：URL 中的 `docId` 固定，`version` 递增
- **用户期望**：新版本只需关注**变动部分**，不重复分析已有的

## 4. 版本差异检测 — 核心设计

### 4.1 数据模型变更

```python
# RequirementDocument 新增字段
class RequirementDocument:
    # 现有字段...
    doc_id: str          # 蓝湖文档ID (跨版本稳定)
    version: str         # 解析后的版本号 (如 "14.2.0")
    parent_id: int | None  # 上一版本的 RequirementDocument.id
    diff_json: str | None  # 结构化差异数据 (JSON)
    diff_status: str     # "initial" | "update" | "patch"
```

```python
# 差异数据结构 (diff_json)
{
    "base_version": "14.1.0",
    "base_job_id": 12,
    "summary": {
        "total_pages": 15,
        "new_pages": 3,
        "modified_pages": 2,
        "unchanged_pages": 9,
        "deleted_pages": 1
    },
    "pages": [
        {
            "page_index": 0,
            "page_name": "首页/推荐流",
            "change_type": "modified",  # new | modified | unchanged | deleted
            "text_similarity": 0.72,     # OCR 文本相似度 (0~1)
            "screenshot_hash": "abc123", # 截图感知哈希 (用于视觉对比)
            "prev_screenshot_hash": "def456",
            "ocr_diff": "新增了「赛事直播」卡片，移除了「广告位」..."
        }
    ]
}
```

### 4.2 差异检测流程

```
Step 1: 解析蓝湖 URL
  └→ 提取 docId + version (如 /updates/14.2.0)

Step 2: 查找上一版本
  └→ SELECT * FROM requirement_documents
      WHERE doc_id = ? AND version < '14.2.0'
      ORDER BY version DESC LIMIT 1
  └→ 找到 v14.1.0 的 RequirementDocument (parent)

Step 3: 采集新版本
  └→ 证据采集 (截图 + OCR) → Job 完成

Step 4: 页面级对比
  ├→ 页码匹配 (按 page_name / page_path 对齐)
  ├→ 文本相似度: difflib.SequenceMatcher(prev_ocr, curr_ocr).ratio()
  │   └→ > 0.90 → unchanged
  │   └→ 0.50~0.90 → modified
  │   └→ < 0.50 or no match → new/deleted
  └→ 截图感知哈希: average_hash(prev_screenshot) vs average_hash(curr_screenshot)
      └→ 汉明距离 < 阈值 → 视觉未变 (辅助判定)

Step 5: 生成 diff_json
  └→ 保存到 RequirementDocument.diff_json
  └→ 设置 diff_status = "update"

Step 6: 后续流程仅处理变动部分
  ├→ 功能拆分: AI 只分析 changed_pages 的 OCR 文本
  ├→ 上下文: 传入 unchanged 页面的摘要 (不做详细拆分)
  └→ 生成用例: 只生成 new + modified 功能点的用例
```

## 6. AI 增强策略

### 6.1 功能拆分 Prompt 改进

```
当前: 只传 OCR 文本 → 信息丢失严重

改进: 传入多模态上下文
  ├─ OCR 文本 (完整)
  ├─ 页面截图描述 (视觉模型生成，或人工标注的页面结构)
  ├─ 版本变动标记 (哪些页面是新的/修改的)
  ├─ 上一版本的功能拆分结果 (作为参考)
  └─ 指令: "仅对标记为 new/modified 的页面做功能拆分，
             unchanged 页面的功能点从上一版本继承"
```

### 6.2 截图描述生成 (可选，Phase 2)

```
方案A: 视觉模型 (如有可用服务)
  截图 → vision API → "这是首页，顶部有搜索栏和Banner轮播，
  中间有5个赛事卡片，底部有Tab导航..."

方案B: 结构化页面分析 (纯工程方案)
  - 分析 DOM/组件树 → 提取页面结构
  - 检测交互元素 (按钮、输入框、列表)
  - 生成页面描述 JSON

方案C: 人工标注 (MVP 方案)
  - 页面列表允许用户添加文字描述
  - 描述作为 AI prompt 补充
```

## 8. 技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 版本检测 | URL 解析 + DB 查询 | 蓝湖 URL 格式固定 `/updates/{version}` |
| 文本差异 | difflib.SequenceMatcher | Python 标准库，无需额外依赖 |
| 截图对比 | 感知哈希 (average_hash / phash) | Pillow 实现，轻量 |
| 进度轮询 | 前端 setInterval 3s | 当前已有 heartbeat 机制 |
| 功能继承 | 直接复制上一版本的功能点 + 用例 | 避免 AI 重复生成，减少成本 |
| 截图预览 | 后端图片代理 + 前端 `<img>` | 已有 `download_asset` API |
