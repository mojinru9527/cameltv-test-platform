# Batch 27 — PRD Summary
> **Product (🟦)** | Date: 2026-07-22
> Status: Draft | Updated: 2026-07-22 (v1.3 — configures 关系 + 附件结构化 + 全局导航 + 降级提取)

## 1. 问题陈述

### 痛点

**1.1 版本全景缺失**：当前需求管理和知识中心无法呈现一个版本的"完整面貌"。体育平台每个版本的完整功能 = 用户端需求（蓝湖用户原型.rp）+ 运营后台需求（蓝湖运营后台.rp），但两端在平台中完全独立存储，没有任何关联。

实际案例：用户端 v3.0.0 新增"资讯"模块 → 运营后台 v4.0.0 需配套"资讯分类"+"资讯列表"两个功能。当前平台无法表达这种跨端关联，测试人员需要手动翻阅两份蓝湖文档才能理解完整需求。

**1.2 知识图谱扁平化**：当前图谱节点类型为 api/field/requirement/test_case/defect，是"技术视角"的扁平分类。用户（测试人员/产品经理）的思维模型是"项目球"：项目 → 版本 → 平台(APP/PC/WEB) → 模块 → 页面 → 功能点。这种层级导航在蓝湖中通过"更新日志→文件夹→页面"自然实现，但在平台中完全缺失。

**1.3 蓝湖文档结构与平台脱节**：蓝湖的核心导航结构——「更新日志」页面（版本号+更新时间+更新内容+页面跳转按钮）——在平台中没有对应的一等实体。版本号只是 `RequirementDocument` 上的一个字符串字段，无法表达版本间的模块演化关系。

**1.4 RAG 与 Wiki 缺乏共同基线**：蓝湖内容导入后分别进入 RAG 知识库和 Wiki，两边独立运行。当 RAG 检索结果与 Wiki 文档不一致时，无法判断哪边是正确的——因为没有共同的"蓝湖原始结构"作为参照基准。

**1.5 🆕 重复导入浪费**：v2+ 版本导入时，当前流程会把整个蓝湖文档重新提取一遍，包括 v1 已经处理过的模块和页面。实际上只需要提取本版本的变更内容——未变更模块无需重新拆分和生成用例。

**1.6 🆕 截图信息丢失**：蓝湖证据包采集了每个页面的截图，但这些截图在导入后没有被保留到模块/页面层级。有些交互（动效、弹窗、状态切换）仅在截图中可见，OCR 文本无法表达。

**1.7 🆕 用例与模块脱节**：功能用例、接口用例、自动化用例生成后与需求模块没有图谱关联。在知识图谱中无法看到"这个模块有哪些测试用例"，也无法从用例反查"这个用例测的是哪个版本的哪个模块"。

**1.8 🆕 v1.2 页面跳转交互丢失**：蓝湖原型中的交互热点（可点击区域、按钮、导航 Tab）构成页面间的跳转关系。例如资讯列表页→搜索页→结果页，底部 Tab 切换等。这些交互链路是需求的组成部分，但在文本提取时完全丢失。AI 生成测试用例时也无法覆盖页面导航流程。

**1.9 🆕 v1.2 说明附件稀疏性**：说明附件（广告位系统、银钻系统、UGC 功能概述等）仅覆盖少数大模块，非每个模块都有。当前设计隐含假设所有模块都有附件，需要修正为附件可选模式。

**1.10 🆕 v1.2 知识中心内容归属混乱**：「项目知识」Tab 混入了平台研发内容（Agent Team 开发日志等）。应明确分工：项目知识 = 体育平台业务知识；平台研发 = 测试平台自身研发资料。

**1.11 🆕 v1.3 跨系统配置链路缺失**：用户端功能的展示内容往往由运营后台配置控制。例如资讯列表的"分类Tab"展示哪些分类由运营后台「资讯分类配置」决定。这种"配置→生效"的跨系统链路在平台中没有被建模，测试时无法评估配置变更的影响面。

**1.12 🆕 v1.3 附件内容未结构化**：说明附件（广告位系统.docx 等）内部包含丰富的功能点、业务规则和流程图，但当前只作为二进制文件挂载到模块上。如果能解析附件内容并提取结构化信息，需求的完整度能再上一个台阶。

**1.13 🆕 v1.3 页面交互提取缺乏降级方案**：v1.2 中页面交互提取仅依赖蓝湖 HTML DOM 抓取和 AI 多模态截图分析两个路径。如果两个路径都不可用（蓝湖 DOM 不可访问、AI 准确率低），整个交互链路功能将不可用。需要补充降级方案。

### 证据

- 蓝湖两份 .rp 文档合计包含 30+ 版本迭代记录
- 用户端覆盖 3 个平台（APP/PC/WEB）+ 说明附件
- 运营后台独立版本号体系，与用户端无必然对应关系
- 当前 `RequirementDocument` 表仅 1 条记录对应 1 份蓝湖导入，无层级结构

## 2. 成功指标

| 指标 | 基线 | 目标 | 测量窗口 |
|------|------|------|---------|
| 版本全景视图可用性 | 无此功能 | 能在一页内看到任一版本的用户端+运营后台完整模块清单 | 上线后 1 周 |
| 图谱层级深度 | 扁平 5 种实体类型 | ≥4 层层级关系（项目→版本→平台→模块→页面） | 上线后立即 |
| 蓝湖→Wiki 同步基线 | 无 | 每次蓝湖导入自动生成 Wiki Raw Source，70%+ 页面有对应条目 | 上线后 2 周 |
| 需求追溯完整性 | 版本号与模块无关联 | 任一模块可追溯其所有版本的变更历史 | 上线后 1 月 |

## 3. 非目标（本次不做）

- **自动从蓝湖抓取更新日志**：仍依赖 lanhu-mcp 手动提取，不做定时自动同步
- **蓝湖原型页面的像素级还原**：平台展示的是结构化需求信息，非原型预览
- **运营后台与用户端的自动关联推断**：关联关系由人工维护（或 AI 辅助建议+人工确认），不做全自动推导
- **知识图谱 3D 可视化**：保持 2D vis-network，不做 Three.js 3D 球体
- **历史数据自动迁移**：新模型上线后，存量数据通过手动/脚本补充层级信息，不做全量自动迁移

## 4. 用户故事 + 验收标准

### US-1: 版本全景视图
**As a** 测试工程师  
**I want** 在需求管理页面看到一个版本的完整面貌（用户端模块 + 运营后台模块 + 说明附件）  
**So that** 我能一次性理解该版本的全部测试范围，无需手动翻阅两份蓝湖文档

**验收标准**：
- Given 已导入用户端 v3.0.0 和运营后台 v4.0.0 的需求文档
- When 我打开版本全景视图，选择用户端 v3.0.0
- Then 我应看到：该版本的用户端模块清单（分 APP/PC/WEB 三列）、关联的运营后台模块、说明附件链接、以及每个模块下的页面列表

### US-2: 知识图谱「项目球」
**As a** 产品经理  
**I want** 在知识图谱中看到一个层级化的项目结构图  
**So that** 我能直观理解项目包含哪些版本、每个版本在不同端有哪些模块、模块下有哪些页面

**验收标准**：
- Given 已建立版本/模块/页面的实体和关系
- When 我打开知识图谱 Tab，选择「项目球」视图
- Then 图谱以层级方式展示：中心节点（项目名）→ 第二层（版本）→ 第三层（平台/端）→ 第四层（模块）→ 第五层（页面）
- And 点击任一节点可展开/折叠其子节点
- And 运营后台模块与对应用户端模块之间有跨端关联边

### US-3: 蓝湖结构同步到 Wiki 基线
**As a** 测试工程师  
**I want** 蓝湖导入的内容自动在 Wiki 知识库中创建对应的目录结构和页面条目  
**So that** RAG 和 Wiki 有共同的数据源基线，我能通过差异对比发现知识不一致

**验收标准**：
- Given 一份蓝湖 .rp 文档被证据采集并导入
- When 导入完成
- Then Wiki 中自动创建对应的目录结构（更新日志→版本条目→文件夹→页面）
- And 每个页面的 OCR 文本作为 Wiki 页面内容存储
- And 在「知识差异对比」Tab 可对比 RAG 切片和 Wiki 页面的差异

### US-4: 模块版本演化追踪
**As a** 测试工程师  
**I want** 看到一个模块在所有版本中的变化历史  
**So that** 我能理解模块的演化路径，识别回归测试范围

**验收标准**：
- Given 模块「资讯」在 v1.0/v2.0/v3.0 三个版本中都有更新
- When 我查看模块「资讯」的详情
- Then 我应看到时间线：v1.0 新增了哪些页面/功能点 → v2.0 修改了什么 → v3.0 的当前状态
- And 每个版本节点可点击跳转到对应的需求文档

### 🆕 US-5: 增量版本导入（v2+ 仅取差异）
**As a** 测试工程师  
**I want** 提交 v2+ 版本需求时，系统自动识别与上一版本的差异，只提取变更的模块和页面  
**So that** 我不需要重复处理未变更的模块，功能拆分和用例生成只针对本版本的实际变更

**验收标准**：
- Given 已存在 v1.0 基线版本的完整模块树
- When 我导入 v2.0 版本需求，选择父版本为 v1.0
- Then 系统自动运行版本 Diff，产出差异报告（新增模块 / 修改模块 / 删除模块 / 未变更模块）
- And 我对 Diff 结果进行人工审核确认
- And 确认后，系统仅提取变更模块的页面内容和截图
- And 未变更模块**不进行**功能拆分和用例生成
- And 修改模块的功能拆分以父模块的已有功能作为上下文
- Given 一个模块在 v2.0 中未被任何变更触及
- When v2.0 的功能拆分和用例生成完成
- Then 该模块不产生任何新的功能点或测试用例

**As a** 测试工程师  
**I want** 在提取需求文档时，不仅保留文案，还保留蓝湖中对应页面的截图

### 🆕 US-7: 测试用例与模块图谱关联
**As a** 测试工程师  
**I want** 在知识图谱「项目球」中看到每个模块/页面关联的测试用例（功能/API/自动化）

**验收标准**：
- Given 模块「资讯」已生成 8 条功能用例、5 条接口用例、2 条自动化用例

### 🆕 US-8: 页面交互跳转链路保留 (v1.2/v1.3)
**As a** 测试工程师
**I want** 从蓝湖提取需求时，不仅保留文案和截图，还要保留页面间的交互跳转关系（点击按钮→跳转目标页面）
**So that** 测试范围能覆盖页面导航流程，AI 能生成导航链路测试用例

**验收标准**：
- Given 蓝湖原型中存在从「资讯列表页」到「搜索页」「资讯详情页」的交互跳转
- When AI 分析页面时
- Then 自动识别页面内的交互热点（按钮、链接、Tab、导航栏）
- And 生成 `page_interactions` JSON 记录每条跳转的触发方式、目标页面、交互类型
- And 自动区分 `dynamic_filter` 类型（目标由后台配置决定）和普通 `navigation`
- And 在版本全景视图中，页面节点显示 outgoing/incoming 跳转关系图
- And 在知识图谱中展示 `navigates_to` 虚线边（默认隐藏，可切换显示）
- And AI 生成测试用例时，额外生成导航链路验证用例
- Given 同一交互（如底部Tab）出现在 >80% 的页面中
- When auto-build 运行时
- Then 该交互自动归类为 `global_navigation`，提升到 ReleaseBundle 层级存储
- And 不从每个页面单独存储，避免冗余
- Given 页面交互蓝湖 DOM 未抓取到，且 AI 多模态也失败
- When 触发降级提取
- Then CV 启发式检测（OpenCV 识别常见 UI 模式）作为自动兜底
- And 仍无法识别时，提供截图标注 UI 供测试工程师手动框选
- Given 运营后台页面存在导航关系（如用户列表→用户详情→编辑用户）
- When 提取运营后台需求时
- Then 同样生成 `page_interactions`，运营后台与用户端使用相同的交互模型

## 5. 技术考量

### 依赖
- 蓝湖证据采集管道（已落地，`lanhu-evidence-pack-pipeline`）
- 知识图谱实体/关系表（已建表，待填充）
- Wiki 知识库导入服务（已有 `wiki_import_service`）

### 已知风险
- **实体爆炸**：如果每个页面/功能点都建实体节点，图谱可能过于密集。→ 策略：默认折叠到模块层级，按需展开
- **版本号不一致**：用户端和运营后台版本号独立，如何表达"v3.0 用户端 + v4.0 运营后台 = 完整版本"？→ 引入 `ReleaseBundle` 概念，一个发布包聚合两端版本
- **存量数据空白**：历史导入的蓝湖文档没有层级信息。→ 提供"补充层级"手动编辑功能 + AI 辅助回填
- 🆕 **Diff 准确率**：AI 在模块重命名/合并/拆分等场景下可能误判。→ Phase A 规则引擎优先，Phase B AI 兜底 + 人工审核
- 🆕 **基线完整性依赖**：v2+ 增量导入的正确性完全依赖 v1 基线的完整。如果 v1 遗漏模块，后续所有版本的 diff 都会漏判。→ v1 基线导入后强制人工逐项审核
- 🆕 **截图存储成本**：每个蓝湖页面 1-3 张截图，30 版本 × 50 页面 = 1500+ 截图。→ 仅存储 URL 引用（蓝湖 OSS），不复制文件
- 🆕 v1.2 **页面跳转识别准确率**：AI 从截图中识别交互热点的准确率有限（可能漏识别、误识别）。→ 优先使用蓝湖原型中的 page-link 数据；AI 识别结果作为补充；支持人工审核和手动编辑
- 🆕 v1.2 **跳转目标未匹配**：`target_page` 名称可能与已知页面节点无法精确匹配。→ auto-build 时输出未匹配列表，提示用户手动关联或创建页面节点
- 🆕 v1.3 **交互提取全部路径不可用**：蓝湖 DOM 抓取 + AI 截图识别 + CV 检测全部失败。→ 提供截图标注 UI 作为最终兜底，确保功能不空白
- 🆕 v1.3 **全局导航阈值边界**：某个交互出现在 79% 的页面中（刚好低于 80% 阈值），被当作普通交互逐页存储。→ 阈值可配置（默认 80%），且支持手动将交互提升为全局导航
- 🆕 v1.3 **附件格式兼容性**：说明附件可能为 .docx / .pdf / .xlsx / .md 多种格式。→ 优先支持 .docx + .pdf（覆盖当前已知附件格式），其他格式降级为仅文件名展示
- 🆕 v1.3 **configures 关系提取准确率**：AI 从模块描述中推断配置控制关系可能不准确。→ AI 建议 + 人工确认模式，低置信度不自动建立关系
- 🆕 v1.3 **知识中心存量迁移风险**：自动分类可能误判。→ 自动规则 + 人工抽检 + CSV 快照回滚
