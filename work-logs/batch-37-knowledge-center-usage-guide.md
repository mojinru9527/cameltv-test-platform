# 知识中心操作文档 — CamelTv 测试平台 v2

> **编写日期**: 2026-07-23 | **适用版本**: v2.1.0+

---

## 1. 概述

知识中心是测试平台 v2 的知识管理中枢，采用 **PARA 方法论**（Projects / Areas / Resources / Archives）组织知识资产。它承担两个核心职责：

| 知识域 | 用途 | 数据来源 |
|--------|------|---------|
| **项目知识** (Project) | 体育项目（CamelTv）的需求文档、接口定义、测试用例 | 需求导入 / OpenAPI 导入 / 用例沉淀 |
| **平台研发** (Platform) | 测试平台自身的研发知识（设计决策、踩坑记录、问题模式） | Agent Team 工件 / AI 审核台 |

**访问路径**: 登录 → 左侧导航「知识」→ 知识中心

---

## 2. 模块功能说明

### 2.1 概览 (Overview)

| 属性 | 说明 |
|------|------|
| **是什么** | 知识中心仪表盘，统计知识源数量、保鲜度、最近更新等 |
| **用在哪里** | 管理员/测试同学快速了解知识库健康状态 |
| **怎么用** | 进入知识中心默认显示概览。关注「待关注」和「过期」标记的知识源，及时更新 |

**举例**: 如果概览显示某个需求文档的保鲜度降到 40%（「过期」），意味着这份需求可能已经与当前产品版本脱节，测试同学应更新或用新版替换。

### 2.2 项目知识 (Project)

| 属性 | 说明 |
|------|------|
| **是什么** | 展示与体育项目相关的知识源：需求文档、接口文档 (OpenAPI)、测试用例 |
| **用在哪里** | 测试同学撰写用例时检索需求，接口测试时查阅接口定义 |
| **怎么用** | 按批次浏览知识源列表，点击任意条目查看详情（元数据 + 原始内容 + 切片列表） |

**举例**:
1. 测试同学需要编写「用户端 14.1.0」版本的测试用例
2. 在知识中心 → 项目知识 → 找到「用户端 14.1.0 需求规格说明书」批次
3. 点击查看，浏览切片中的功能点描述
4. 用切片内容作为 AI 生成用例的输入参考

**怎么延伸**:
- 在需求管理模块导入需求文档后，知识自动沉淀到项目知识
- 在 API 测试模块导入 Swagger/OpenAPI 文档后，接口定义自动沉淀
- 后期可实现：点击知识切片直接跳转到关联的用例/接口/需求

### 2.3 平台研发 (Platform)

| 属性 | 说明 |
|------|------|
| **是什么** | 测试平台自身的研发知识库：设计决策、问题模式、审查报告 |
| **用在哪里** | 平台开发者/AI Agent 编码时检索历史问题和最佳实践 |
| **怎么用** | 按 PARA 分类（问题模式/设计规范/归档）折叠浏览，点击条目查看详情 |

**举例**:
1. Agent Team 在 Dev 部门编码前，检索平台研发知识库
2. 发现历史记录：「`R.err()` 方法在 `common.py` 中未定义，多处调用会崩溃」
3. Agent 据此避开已知坑，在代码中检查是否还有其他缺失方法

**怎么延伸**:
- 每次 Agent Team 批次的 Leader 审查后，新发现的模式自动入库
- 可配置知识保鲜策略，自动标记过期知识

### 2.4 检索 (Search)

| 属性 | 说明 |
|------|------|
| **是什么** | 全知识库统一检索，支持混合（关键词+向量）、关键词、向量语义三种模式 |
| **用在哪里** | 需要快速查找某个知识点，但不记得在哪个模块 |
| **怎么用** | 在搜索栏输入关键词 → 选择检索模式 → 点击搜索 |

**举例**:
1. 测试同学想知道「支付」相关的所有测试用例和需求
2. 输入「支付」→ 选择「混合」模式 → 搜索
3. 返回结果包含：需求文档中支付章节、接口用例中支付 API、历史执行记录中支付相关失败

**怎么延伸**:
- 支持按知识域/类型/时间范围过滤
- 搜索结果可直接链接到来源（需求/用例/缺陷）

### 2.5 知识源 (Sources)

| 属性 | 说明 |
|------|------|
| **是什么** | 知识源管理页面，按来源类型分类（需求/接口导入/测试用例/缺陷/执行失败/手动） |
| **用在哪里** | 管理员管理知识库数据的入口 |
| **怎么用** | 按来源类型筛选 → 点击条目查看详情 → 可验证（保鲜度刷新） |

**举例**:
1. 有 10 个需求文档导入了知识库，但不确定哪些是最新的
2. 进入知识源 → 筛选「需求」→ 查看保鲜度评分 → 手动验证到期文档

### 2.6 AI 审核台 (Artifacts)

| 属性 | 说明 |
|------|------|
| **是什么** | AI 生成的工件审核台：Agent Team 产出的 PRD/PM/Design 等工件经 AI 审核后，可在此确认入库 |
| **用在哪里** | Agent Team Leader 审查前，先由 AI 自动审核工件质量 |
| **怎么用** | 查看待审核工件 → 审查 AI 审核意见 → 批准/驳回 → 批准后工件入库为平台研发知识 |

### 2.7 图谱 (Graph)

| 属性 | 说明 |
|------|------|
| **是什么** | 知识实体关系图谱，可视化展示需求-用例-缺陷-接口之间的关联 |
| **用在哪里** | 追溯某个需求覆盖了多少用例、发现了多少缺陷 |
| **怎么用** | 选择知识域 → 查看节点和关联边 → 点击节点查看详情 → 支持关系演化 |

### 2.8 实体 (Entities)

| 属性 | 说明 |
|------|------|
| **是什么** | 从知识库中提取的结构化实体列表（如功能点、API 端点、缺陷类型） |
| **用在哪里** | 快速浏览知识库中的核心概念 |

### 2.9 迭代 (Iterations)

| 属性 | 说明 |
|------|------|
| **是什么** | 知识版本迭代管理：快照对比、差异分析 |
| **用在哪里** | 追踪需求变更对用例的影响 |
| **怎么用** | 选择迭代 A → 选择迭代 B → 查看差异 |

**举例**:
1. 「用户端 14.0.0」升级到「14.1.0」
2. 在迭代 Tab 选择两个版本的需求知识快照
3. 对比结果：新增了 3 个功能点、修改了 2 个接口、删除了 1 个模块
4. 测试同学据此更新测试计划

### 2.10 Wiki 知识库

| 属性 | 说明 |
|------|------|
| **是什么** | 类 Wiki 的知识编辑和管理功能，支持知识差异对比 |
| **用在哪里** | 团队维护长文档型知识资产 |

### 2.11 Skills / 项目球

| 属性 | 说明 |
|------|------|
| **是什么** | Skills：管理的 AI 技能列表；项目球：多项目知识的可视化球体 |

## 3. 典型使用场景

### 场景 A：测试同学编写新版本用例

```
1. 需求管理 → 导入「用户端 14.1.0 需求规格说明书」
2. 知识中心 → 项目知识 → 确认需求文档已入库
3. 知识中心 → 检索 → 搜索「14.1.0」获取所有相关切片
4. 用例服务 → AI 生成 → 选择知识源作为输入
5. 生成用例 → 审阅 → 入库
```

### 场景 B：接口测试同学查阅 API 定义

```
1. API 测试 → 导入 Swagger 文档
2. 知识中心 → 项目知识 → 找到接口导入的知识源
3. 知识中心 → 图谱 → 查看接口与用例/需求的关联
4. API 测试 → 基于知识切片编写接口调用
```

### 场景 C：平台开发者避坑

```
1. 开发新功能前 → 知识中心 → 平台研发 → 检索模块名
2. 逐个查阅历史问题模式（area 分类）
3. 参考设计规范（resource 分类）
4. 编码时主动避开已知陷阱
```

## 4. 数据保鲜机制

| 分数 | 等级 | 颜色 | 建议操作 |
|------|------|------|---------|
| ≥80% | 新鲜 | 🟢 绿色 | 正常使用 |
| 40-80% | 待关注 | 🟡 黄色 | 尽快验证 |
| <40% | 过期 | 🔴 红色 | 更新或归档 |

知识保鲜评分基于：最后验证时间、内容来源更新频率、下游用例执行通过率等因子综合计算。

---

## 5. 延伸建议

1. **需求联动**: 知识中心的需求切片 → 一键跳转到需求管理模块
2. **用例联动**: 知识切片的接口定义 → 一键生成接口用例
3. **自动保鲜**: 上游需求文档/接口文档变更时，自动标记关联知识为「待关注」
4. **报表导出**: 支持导出某模块的全部知识为 PDF/Markdown
5. **团队协作**: 知识源支持评论/标注，团队成员可补充使用心得
