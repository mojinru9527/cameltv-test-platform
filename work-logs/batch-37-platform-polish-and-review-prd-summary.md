# Batch 37 — PRD Summary

> **Product (🟦)** | Date: 2026-07-23 | Status: Draft

## 1. 问题陈述

用户（资深测试工程师 + 项目管理视角）对测试平台 v2 初版进行重度使用后，发现以下核心问题：

1. **缺乏全面质量审查**：平台从未做过系统级的代码、功能、UI 三合一审查。历史缺陷反复出现（Agent Team 逻辑漏洞修了好几次仍回归），说明缺少结构性自查机制。

2. **知识中心偏离设计初衷**：当前「项目知识」Tab 存储的是 Agent Team 的 PRD 摘要、PM 计划等平台研发产物，而非体育项目的需求文档、接口文档、测试用例。这与知识中心的定位（为使用测试平台的测试同学提供项目知识检索）完全不符。

3. **知识中心展示体验差**：模块列表点击后的弹窗太小（`max-w-5xl w-[95vw]` 实际内容展示空间不足），切片回显内容展示不全。

4. **知识中心功能缺乏验收**：没有用真实的用户端需求文档、运营后台需求文档和生产环境地址进行功能验收，也没有使用文档说明知识中心各模块怎么用。

5. **平台初版未通过资深视角验证**：用户以资深测试工程师 + 项目管理视角重度使用后，需要判断初版是否符合预期，是否存在功能缺失或遗漏，以及未来延伸方向。

6. **模块间缺乏联动**：需求文档 → AI 用例生成 → 接口测试 → UI 自动化的链路是断裂的。测试同学不能从一个需求文档出发，联动多个功能模块，无法节省时间、提高效率。

7. **未完成模块暴露在导航中**：版本测试任务、缺陷管理、测试数据集、集成配置等尚未完成开发的模块仍然出现在导航菜单中，给用户造成困惑。

8. **全局字体偏小**：平台各处字体（正文 `text-sm` 14px、标题 `h4` 16px、侧边栏 `text-sm` 14px 等）整体偏小，影响可读性和使用体验。

## 3. 非目标（本次不做）

- **模块联动全链路实现**（任务 6）：本 Batch 只做设计规划，具体编码实现放到后续 Batch。原因：涉及需求→用例→接口→UI 四大模块改造，范围太大。
- **知识中心后端数据清洗**：不删除已有的平台研发知识数据，只在前端展示层面区分「项目知识」和「平台研发」数据源。
- **新模块开发**：不新增功能模块，只做现有模块的审查、修复和优化。
- **蓝湖 MCP 数据自动同步**：任务 4/5 的手动验收不触发蓝湖自动抓取，仅使用用户提供的链接作为参考。
- **生产环境部署**：本 Batch 所有改动在本地验证，不涉及 test/staging/prod 部署。

## 4. 用户故事 + 验收标准

### US-1: 测试平台全面代码/功能/UI 审查

**As a** 测试平台负责人
**I want** 对测试平台 v2 进行代码、功能和 UI 三合一审查
**So that** 我能清楚知道平台当前存在哪些质量问题、哪些需要优化

**验收**：
- Given 测试平台 v2 代码库 / When 执行审查 / Then 产出一份审查报告，包含：
  - 代码层面：重复代码、性能隐患、安全问题、未处理异常、类型安全
  - 功能层面：各模块功能完整性、边界情况处理、错误处理
  - UI 层面：视觉一致性、响应式表现、交互体验问题
  - 每项发现标注严重级别（P0–P3）和所在文件:行号

### US-2: Agent Team 逻辑审查

**As a** Agent Team 维护者
**I want** 审查 Agent Team 的代码实现是否存在逻辑漏洞
**So that** 同样的问题不再来来回回修复多次

**验收**：
- Given Agent Team 技能文件和脚本 / When 逐文件审查逻辑 / Then：
  - 检出所有可能导致重复修复的逻辑漏洞
  - 每个漏洞标注根因（为什么之前修复没覆盖）
  - 提出修复建议

### US-3: 知识中心数据源修正 + 弹窗尺寸优化

**As a** 测试同学
**I want** 在知识中心看到体育项目的需求、接口、用例，而不是平台研发的 PRD/PM 工件
**So that** 我能用知识中心检索项目知识，辅助测试工作

**As a** 测试同学
**I want** 知识中心模块列表的详情弹窗足够大
**So that** 回显内容能完整展示，不需要反复滚动

**验收**：
- Given 知识中心「项目知识」Tab / When 浏览数据源 / Then：
  - 数据仅来源于项目知识库（需求文档/接口文档/用例），不含 Agent Team 工件
  - 如果当前数据库尚无项目知识数据，显示空状态引导「导入需求文档和接口文档后，项目知识将在这里展示」
- Given 知识中心模块列表某条记录 / When 点击打开弹窗 / Then：
  - 弹窗宽度至少占视口 85%，高度至少占视口 90%
  - 切片内容区域高度足够展示完整内容
  - 在 1920×1080 分辨率下弹窗内容无需横向滚动

### US-4: 知识中心功能验收 + 操作文档

**As a** 知识中心使用者
**I want** 一份操作文档说明知识中心各模块是什么、怎么用、怎么延伸
**So that** 我能快速上手使用知识中心

**验收**：
- Given 用户端需求文档链接、运营后台需求文档链接、生产地址 / When 编写操作文档 / Then：
  - 文档包含：每个模块的功能说明、使用场景、操作步骤、举例说明
  - 举例须基于真实体育项目数据（CamelTv 体育平台）
  - 包含延伸使用建议

### US-5: 平台初版资深视角验证

**As a** 资深测试工程师 + 项目管理者
**I want** 重度使用测试平台所有主要模块，对照需求文档和生产环境验证
**So that** 我能判断初版是否符合预期，明确缺失和遗漏

**验收**：
- Given 用户端需求文档、运营后台需求文档、生产环境地址 / When 逐模块重度使用 / Then：
  - 涵盖：知识中心、需求文档、用例服务、接口测试、UI 自动化测试 + 其他次要模块
  - 每个模块验证：功能完整性、是否符合需求、是否存在遗漏
  - 产出功能符合度矩阵 + 缺失/遗漏清单 + 延伸开发建议

### US-6: 模块联动设计规划

**As a** 测试同学
**I want** 导入一份需求文档后能自动做功能拆分、AI 生成用例，并延伸到接口测试和 UI 自动化
**So that** 一个需求文档能联动多个功能，节省时间，提高效率

**验收**：
- Given 当前平台模块 / When 设计联动方案 / Then：
  - 产出模块联动架构设计文档（需求→用例→接口测试→UI 自动化）
  - 明确接口用例依赖 Swagger 文档上传的前提条件
  - 明确 UI 自动化关联上一个版本已上线功能的范围
  - 此用户故事本 Batch 只做设计，不编码实现

### US-7: 隐藏未完成模块

**As a** 平台用户
**I want** 导航菜单中不显示尚未开发完成的模块
**So that** 我不会被未完成功能困惑，界面更整洁

**验收**：
- Given 当前导航菜单 / When 加载 / Then：
  - 「版本测试任务」不出现在导航菜单
  - 「缺陷管理」不出现在导航菜单
  - 「测试数据集」不出现在导航菜单
  - 「集成配置」不出现在导航菜单
  - 已完成的模块（工作台、需求管理、用例服务、测试计划、运行中心、API 测试、UI 自动化、报告中心等）保持可见

### US-8: 全局字体放大

**As a** 平台用户
**I want** 平台所有地方的字体都大一些
**So that** 长时间使用不费眼，阅读体验更好

**验收**：
- Given 平台全局样式 / When 调整字体 / Then：
  - 正文基础字号从 14px 提升到 15px（Tailwind `text-base` 级别调整）
  - 侧边栏菜单项字号从 `text-sm` 提升到 `text-base`（15px）
  - 标题 h1–h4 各级别等比例放大
  - 表格内容字号从 `text-sm` 提升到 `text-base`
  - 卡片标题和描述文字适当放大
  - 不影响布局和响应式表现
  - 用户主观确认「看起来舒服了」

## 5. 技术考量

| 依赖/风险 | 说明 |
|-----------|------|
| 蓝湖链接访问 | 任务 4/5 依赖用户提供的蓝湖链接获取需求文档内容 |
| 生产环境 | 任务 5 验收需要访问 `https://www.camel1.tv/` |
| 知识中心数据源 | 如果数据库中没有体育项目知识数据，需要提供导入渠道 |
| 导航菜单隐藏 | 菜单由后端 `fetchMenus()` API 动态返回，需后端配合修改 |
| 字体放大 | 涉及 `globals.css` 和 Tailwind 基础字号配置，影响面广需逐个页面回归 |
| Agent Team 逻辑 | 涉及 `DEPARTMENTS.md`、多个技能文件和脚本，需逐文件审查 |
| 工作树冲突 | Batch 37 工作树已创建于 `feature/batch-37-platform-polish-and-review` |
