# CamelTv 测试平台 · 前后端分离重构方案 (v2.1)

> **文档定位**：把现有 Python 单体测试平台，重构为「前后端独立仓库 + FastAPI 统一后端 + 完整 RBAC 多项目」的标准测试管理平台，对标 MeterSphere v3。
>
> **决策基线**（已拍板）：
> 1. 后端 = **Python / FastAPI + SQLite**（单后端一套技术栈，复用现有全部投入，业务与执行分层不分语言）
> 2. 代码组织 = **前后端两个独立仓库**，各自独立 CI/CD、独立版本、契约对接
> 3. 用户权限 = **完整 RBAC + 多项目隔离**
> 4. 实施节奏 = **重新搭骨架 → 再迁移现有资产**
>
> 阅读约定：✅ 已有可复用 / 🟡 部分有需改造 / 🔴 待新建。
>
> > v2.1 变更：后端由「Java 主干 + Python 引擎」双栈，调整回 **纯 Python（FastAPI + SQLite）单栈**，降低维护门槛、最大化复用现有代码。

---

## 0. 为什么要重构（现状痛点）

| 痛点 | 现状 | 重构后 |
|---|---|---|
| 前后端耦合 | `web-ui/` 和 `server/` 同仓，部署绑死 | 两仓独立，前端/后端团队解耦并行 |
| 无用户体系 | 谁都能改用例、触发 prod 执行，不可追溯 | RBAC + 审计日志，操作可追溯 |
| 单项目硬编码 | 只服务 CamelTv 一个项目 | 多项目隔离，复制配置即接入新项目 |
| 触发逻辑散落 | 手动/定时/CI 三套各写各的 | 统一调度（APScheduler）+ 后台任务 |
| 数据层薄弱 | 裸 `sqlite3` + 散落 SQL | SQLAlchemy ORM，SQLite 起步、可平滑升 Postgres |
| 业务与执行混在一起 | FastAPI 路由里既写 SQL 又跑测试 | 分层：API → Service → 执行引擎 worker |

```
└───────────────────────────┬────────────────────────────────┘

                            │  契约: FastAPI 原生 OpenAPI → 前端生成 TS 类型
┌───────────────────────────▼────────────────────────────────┐
│  仓库 B: cameltv-test-backend  (独立仓库, 纯 Python)          │
│ ┌────────────────────────────────────────────────────────┐ │

│ │  存储: SQLite (data/platform.db, WAL) → 可升 PostgreSQL  │ │
│ ├────────────────────────────────────────────────────────┤ │
│ │  执行引擎 (进程内, 后台任务/线程池)                       │ │
│ │  Playwright(接口) · midscene(UI) · 音视频专项 ·          │ │
│ │  6 件套工具 · RunContext 多环境                          │ │
│ │  调度: APScheduler (定时任务)                            │ │
│ └────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
              │                          │
        ┌─────▼─────┐              ┌─────▼─────┐
        │ 被测环境   │              │ ELK/Kibana │
        │ test/prod │              │ traceId 链路│
        └───────────┘              └───────────┘
```

**关键设计：业务与执行分层，但同在一个 Python 进程/技术栈内。**
- API 层只做参数校验 + 鉴权，调 Service；Service 写库 + 投后台任务；执行引擎在后台线程/进程跑测试，实时回写库。
- 无需 Java、无需 RabbitMQ：内部用 **FastAPI BackgroundTasks / 线程池** 跑执行，用 **APScheduler** 跑定时；并发压力大时再平滑接 Celery + Redis（预留接口，非必须）。

## 2. 仓库与分支策略（你最关心的「代码分支科学合理」）

### 2.1 两仓职责

| 仓库 | 内容 | 技术栈 | 发布物 |
|---|---|---|---|
| `cameltv-test-frontend` | 测试平台 Web UI | React/Vite/TS | Nginx 静态镜像 |
| `cameltv-test-backend` | FastAPI 后端 + 执行引擎 + `deploy/` | FastAPI/SQLAlchemy | 后端镜像 + compose |

> 现有 `test-platform/tools/`（6 件套）、Playwright/midscene 引擎、`config/` 多环境，**全部并入后端仓的执行引擎层**，几乎原样复用。

### 2.2 分支模型（两仓统一用 Git Flow 简化版）

```
main      ──●────────●────────●──    生产，只接受 release/hotfix 合入，打 tag
             ╲        ╱        ╲
develop   ──●─●──●──●─●──●──●──●──    集成分支，feature 汇入
             ╲ ╱    ╲ ╱
feature/*   ●─●      ●─●              单功能分支，从 develop 切出，PR 回 develop
release/*            ●──●            发版冻结、提测、回归
hotfix/*                    ●─●      线上紧急修复，从 main 切，回 main + develop
```

- 分支命名：`feature/用例批量导入`、`fix/计划执行通过率`、`release/v1.2.0`
- 保护规则：`main`/`develop` 禁止直接 push，必须 PR + Code Review + CI 绿灯
- 提交规范：Conventional Commits（`feat:` / `fix:` / `refactor:` / `docs:`）

### 2.3 前后端不脱节的关键：契约先行（FastAPI 原生支持，几乎零成本）

```
后端 FastAPI (Pydantic 模型)
        │  自动暴露 /openapi.json (FastAPI 内建，无需额外工具)
        ▼
   契约文件 (后端仓 CI 导出 docs/openapi.json)
        │  openapi-typescript
        ▼
前端 src/types/api.d.ts (自动生成的 TS 类型)
```

- 后端改 Pydantic 模型 → `/openapi.json` 自动变 → 前端拉取生成类型，**编译期就能发现契约不一致**。
- 前端开发不必等后端：用 OpenAPI 起 Mock（Prism/MSW），并行开发。
- API 版本：统一 `/api/v1` 前缀，破坏性变更升 `/api/v2`。

### 2.4 版本对齐

- 前后端各自语义化版本（SemVer），互不绑定。
- 对外整体能力用「产品版本号」标识（遵循团队约定：取蓝湖「更新日志」手写版本，非系统版本）。
- 部署清单 `deploy/versions.yaml` 记录「某次上线 = 前端 vX + 后端 vY」。

## 4. 数据库设计（SQLAlchemy ORM，SQLite 起步，多项目隔离）

> 核心原则：除「用户/角色/权限」等全局表外，**所有业务表带 `project_id`** 做项目隔离。
> 用 SQLAlchemy 定义模型，Alembic 管迁移；**换数据库只改连接串**（SQLite → PostgreSQL 零改代码）。

### 4.1 系统/权限域

| 表 | 关键字段 | 说明 |
|---|---|---|
| `sys_user` | id, username, password(BCrypt), nickname, email, status, last_login_at | 用户 |
| `sys_role` | id, code, name, data_scope | 角色（全局或项目内） |
| `sys_permission` | id, code, name, type(menu/button/api), parent_id | 权限点/菜单树 |
| `sys_user_role` | user_id, role_id, project_id | 用户在某项目下的角色 |
| `sys_role_permission` | role_id, permission_id | 角色-权限 |
| `sys_project` | id, code, name, owner_id, status, config(JSON) | 项目（多项目隔离主键） |
| `sys_project_member` | project_id, user_id, role_id | 项目成员 |
| `sys_dict` / `sys_dict_item` | type, label, value | 数据字典（优先级/状态等枚举集中管理） |
| `sys_audit_log` | id, user_id, project_id, action, target, detail(JSON), ip, created_at | 审计日志 |

### 4.2 配置/需求域

| 表 | 关键字段 | 说明 |
|---|---|---|
| `env_config` | id, project_id, env(test/prod), base_url, proxy_strategy(JSON), db_dsn, redis, log_source | 环境配置（替代现 config/*.yaml 的 UI 可编辑版） |
| `requirement` | id, project_id, version, role(用户端/运营/管理员), title, content, source(蓝湖URL) | 需求文档（版本号遵循蓝湖更新日志） |
| `requirement_point` | id, requirement_id, point, priority(P0-P3) | 测试点 |
| `requirement_case_link` | requirement_id, case_id | 需求↔用例追溯矩阵 |

### 4.3 用例域

| 表 | 关键字段 | 说明 |
|---|---|---|
| `test_case` | id, project_id, module, title, priority, status, type(api/ui/manual), tags(JSON), preconditions, steps(JSON), expected_result, source, version, creator_id | 用例库（现 901 条迁入此表） |
| `api_case` | id, case_id, method, path, headers(JSON), body(JSON), asserts(JSON), db_check(JSON) | 接口用例执行细节（swagger 生成） |

### 4.4 计划/执行域

| 表 | 关键字段 | 说明 |
|---|---|---|
| `test_plan` | id, project_id, name, env, status, pass_rate, creator_id | 测试计划 |
| `test_plan_case` | plan_id, case_id, sort_order, status, actual_result, executed_at | 计划-用例关联及执行态 |
| `test_plan_run` | id, plan_id, trigger_type(manual/cron/ci), status, started_at, finished_at, summary(JSON), operator_id | 执行记录 |

### 4.5 调度/报告/专项/通知域

| 表 | 关键字段 | 说明 |
|---|---|---|
| `schedule_job` | id, project_id, name, cron, job_type(plan/api/ui/av), target_id, enabled | 定时任务（APScheduler 持久化） |
| `schedule_log` | id, job_id, fire_time, status, run_id | 调度执行日志 |
| `test_report` | id, project_id, run_id, type, total, passed, failed, pass_rate, report_url | 报告 |
| `report_detail` | id, report_id, case_id, status, duration, trace_id, log_url | 用例级结果（trace_id 联动 ELK） |
| `defect` | id, project_id, case_id, title, severity, status, external_id(禅道/Jira) | 缺陷 |
| `av_check_task` | id, project_id, stream_url, protocol(HLS/FLV/WebRTC) | 音视频专项任务 |
| `av_check_metric` | id, task_id, metric(起播时延/卡顿率/音画同步...), value, threshold, pass | 专项指标结果 |
| `notify_channel` | id, project_id, type(钉钉/企微/邮件), webhook, enabled | 通知渠道 |
| `notify_record` | id, channel_id, run_id, content, status | 通知记录 |

### 4.6 数据迁移：旧 SQLite → 新 SQLite(ORM)

现有 `data/platform.db` 的 901 条用例 + 计划 → 写一次性迁移脚本：
- 字段映射：旧 `test_cases` → 新 `test_case`（补 `project_id=CamelTv默认项目`、`creator_id=system`）
- JSON 字段（tags/steps）原样搬运
- 旧 `test_plans/test_plan_items/test_plan_runs` → 新计划三表
- 校验：迁移后行数比对 + 抽样核对；保留旧库备份

## 7. 关键交互：一次「运行测试计划」的全链路（纯进程内，无 MQ）

```
1. 前端  POST /api/v1/plans/{id}/run            (带 JWT, 当前 project_id)
2. FastAPI  鉴权 → 校验权限点 plan:run → 写 test_plan_run(status=running)
            → 记审计日志 → 提交后台任务(线程池/BackgroundTasks) → 立即返回 run_id
3. worker   装配 RunContext(按 env) → 逐条跑 Playwright/midscene
            → 每条实时回写 test_plan_case + report_detail（同库，无需回调 HTTP）
4. worker   全部跑完 → 算 pass_rate → 更新 test_plan_run(passed/failed)
            → 触发 notify（钉钉/企微）→ 失败用例挂 ELK traceId 链接
5. 前端  轮询 GET /api/v1/runs/{run_id}（或 WebSocket）→ 渲染报告 + 一键跳 Kibana
```

> 定时任务：APScheduler 到点 → 直接复用第 2~4 步同一套执行逻辑，`trigger_type=cron`。

## 8. 实施路线图（重新搭骨架 → 再迁移）

| 阶段 | 目标 | 关键产出 | 验收 |
|---|---|---|---|
| **P0 骨架** | 两仓初始化 + 鉴权打通 | 前后端仓库 + CI/CD + SQLAlchemy 模型 + Alembic + 登录/JWT + 项目切换 + OpenAPI→TS 契约链路 | 能登录、切项目、调通一个鉴权接口 |
| **P1 系统域** | RBAC + 多项目 | 用户/角色/权限/项目管理 + 审计日志 + 动态菜单 | 不同角色看到不同菜单，操作留痕 |
| **P2 用例迁移** | 用例服务上线 | 用例 CRUD + 901 条旧库迁移 + 需求追溯 | 旧用例全部可见可查，行数核对一致 |
| **P3 计划+执行** | 跑通核心闭环 | 计划编排 + 后台 worker + 接口用例执行（Playwright） | 建计划→关联用例→运行→出结果 |
| **P4 报告+看板+调度** | 度量与自动化 | 三类报告统一 + 工作台看板 + APScheduler 定时 + 通知中心 | 定时跑回归并推送钉钉 |
| **P5 专项+联动** | 补齐特色能力 | 音视频专项 + ELK/traceId 联动 + 缺陷管理 + UI 自动化(midscene) | 失败用例一键跳 Kibana，音视频指标出报告 |
| **P6 可移植** | 一键部署 | docker-compose 全栈 + 项目接入向导 + 接入文档 | 新项目复制配置包即可起一套 |

## 9. 部署形态（deploy/ 目录，后端仓内）

```yaml
# docker-compose.yml（一键起全栈，极简）
services:
  backend:    # FastAPI (uvicorn/gunicorn)，挂载 sqlite 数据卷 + 报告卷
  frontend:   # Nginx 静态站点 + 反代 /api → backend
  # 可选: redis(若启用 Celery/缓存), wiremock(Mock Server)
```

- 一条命令 `docker-compose up -d` 起全套（对齐既有「可移植 + 一键部署」目标）。
- SQLite 以数据卷挂载，无需独立数据库容器；并发压力大时把 `backend` 的连接串切到 PostgreSQL 容器即可。
- 项目接入向导：`tp init-project <name>` 生成空白项目配置包（环境/域名/凭据/swagger）。

## 10. 风险与权衡（提前说清）

| 风险 | 影响 | 缓解 |
|---|---|---|
| SQLite 并发能力有限 | 多人同时跑执行可能锁库 | WAL 模式 + 执行任务串行化/限流；预留 PostgreSQL 升级路径（仅改连接串） |
| 重写工作量 | 仍需搭骨架 + 迁移 | 比 Java 方案大幅减负；严格 P0→P6 分阶段，每阶段可独立上线 |
| 现有裸 SQL 重构成 ORM | 需重写数据访问层 | 业务逻辑可参考现 `store.py`，迁到 Service/Repository 分层 |
| 执行引擎与 Web 同进程 | 重任务可能拖慢 API | 执行走独立线程池/进程；后续可拆 Celery worker（接口已预留） |
| 数据迁移出错 | 901 条用例丢失 | 迁移脚本 + 行数/抽样双校验 + 保留旧库备份 |
| 前后端契约漂移 | 联调返工 | FastAPI 原生 OpenAPI 自动生成 TS 类型，编译期拦截 |
